# 如何禁止类被继承

## 核心结论

C++11 起直接用 **`final`** 修饰类:`class MyClass final { ... };`,编译器在解析继承关系时发现目标是 final 类,直接报错,**零运行时开销**。C++11 之前需要用**私有构造函数 + 虚继承 + 友元**的组合 hack,复杂且已被淘汰。

`final` 还有第二种用法:**修饰虚函数** `void speak() final;`,禁止派生类重写该虚函数。两者都是**编译期静态检查**。

| 用法 | 语法 | 作用 |
|------|------|------|
| 修饰类 | `class Foo final { ... };` | 禁止任何类继承 `Foo` |
| 修饰虚函数 | `void f() final;` | 禁止派生类重写 `f()` |
| 同时使用 | `void f() override final;` | 重写基类虚函数 + 禁止再重写 |

## final 关键字的底层原理

编译器在解析继承关系或虚函数重写时,检查目标是否标记了 `final`。如果是,直接报编译错误:

- `cannot derive from 'final' base` —— 试图继承 final 类
- `cannot override 'final' function` —— 试图重写 final 虚函数

**不需要任何运行时机制**。这与 Java 的 `final` 完全一致(Java 关键字叫法相同,语义略有差异)。

## final 的性能优化:去虚拟化

编译器知道一个类是 final 或者一个虚函数是 final 后,可以做**去虚拟化(devirtualization)** 优化:

- 因为不可能有派生类重写这个函数,编译器可以在编译期确定调用目标
- 把虚函数调用(通过 vptr 间接跳转)优化成**直接调用**甚至**内联**
- 热路径上频繁调用的虚函数加 `final` 能有明显性能提升(5-15% 不等,看调用频率)

```cpp
class Base {
public:
    virtual int compute(int x) const { return x * 2; }  // 虚函数
};
class Derived final : public Base {  // final 类
    // 编译器对 d.compute(5) 可以去虚拟化
};
Derived d;
auto y = d.compute(5);  // 优化后等价于 y = 10
```

## 什么场景下应该禁止继承

| 场景 | 加 final 的理由 |
|------|----------------|
| 工具类(只有静态方法) | 继承没意义 |
| 资源句柄类(`FileDescriptor`) | 继承可能破坏 RAII 语义 |
| 性能关键类 | 帮助编译器去虚拟化 |
| 策略类 | 通过组合使用而非继承 |
| 第三方库的内部实现类 | 防止用户破坏封装契约 |

**核心原则:组合优于继承**——如果一个类的设计意图是被组合使用而非被继承扩展,就应该加 `final` 明确表达这个意图。

## C++11 之前的替代方案(已淘汰)

核心思路是利用虚继承的特性:**虚继承要求最终派生类直接调用虚基类的构造函数**。

```cpp
// C++98 时代的"模拟 final"方案
class MakeFinal {  // 辅助类
private:
    MakeFinal() {}
    friend class Sealed;
};
class Sealed : virtual private MakeFinal {  // 虚继承 + 私有构造
public:
    Sealed() {}
};
class Try : public Sealed {};  // 编译错误:无法访问 MakeFinal 私有构造函数
```

**为什么能 work**:

1. `Sealed` 虚继承 `MakeFinal`,且声明 `MakeFinal` 为友元,所以 `Sealed` 能访问 `MakeFinal` 的私有构造
2. `Try` 试图继承 `Sealed`,但 `Try` 不是 `MakeFinal` 的友元
3. 虚继承要求最终派生类(`Try`)直接调用 `MakeFinal` 的构造函数
4. `Try` 访问不到 `MakeFinal` 的私有构造 → **编译错误**

这个方案**依赖复杂的继承关系和访问控制**,可读性差、维护成本高,现在完全没必要用了。

## final 和 override 的关系

两者都是 C++11 引入的**上下文关键字**(contextual keyword),只在特定位置有特殊含义,可以作为普通标识符使用。

| 关键字 | 作用 | 解决的问题 |
|--------|------|-----------|
| `override` | 告诉编译器"我确实在重写基类虚函数" | 写错签名编译器不报错,导致隐式隐藏 |
| `final` | 告诉编译器"到此为止,不允许再重写/继承" | 防止下游破坏设计契约 |

**配合使用**表示双重约束:

```cpp
class Base {
public:
    virtual void speak() const = 0;  // 纯虚:派生类必须实现
};
class Derived : public Base {
public:
    void speak() const override final {  // 实现 + 不允许再重写
        std::cout << "Derived speak\n";
    }
};
// class Further : public Derived {
// public:
//     void speak() const override {}  // 错误:cannot override 'final' function
// };
```

## 面试高频追问

**Q1: final 类是否还可以被聚合(aggregate)?**
可以。`final` 只禁止派生,不影响作为值类型的拷贝/移动构造和聚合初始化:

```cpp
class Point final {
public:
    int x, y;
};
Point p{1, 2};  // OK,聚合初始化
```

**Q2: final 和 constexpr 能同时用吗?**
可以,且通常一起用表示"编译期常量 + 不可派生":

```cpp
class MathConst final {
public:
    static constexpr double pi = 3.14159;
};
```

**Q3: 抽象类能 final 吗?**
可以。`final` 修饰类,和是否包含纯虚函数无关。`final` 抽象类表示"这个接口只能被这几个派生类实现,不允许用户再扩展"。

**Q4: final 虚函数还能被基类指针调用吗?**
可以。`final` 只禁止派生类**重写**,不禁止基类**调用**:

```cpp
class Base { public: virtual void f() final { std::cout << "base\n"; } };
class D : public Base { /* 不能重写 f() */ };
Base* p = new D();
p->f();  // 合法,调用 Base::f()
```
