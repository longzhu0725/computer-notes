# C++ 智能指针选型

## 核心结论

三种智能指针对应三种**所有权模型**:`unique_ptr` 独占、`shared_ptr` 共享、`weak_ptr` 观察。**选型原则**:能用 `unique_ptr` 就不用 `shared_ptr`,需要打破循环引用才用 `weak_ptr`。

| 智能指针 | 所有权 | 性能 | 典型场景 |
|---------|--------|------|---------|
| `unique_ptr` | 独占 | **零开销**(等同裸指针) | 90% 场景:函数局部资源、类成员、工厂返回值 |
| `shared_ptr` | 共享(引用计数) | 控制块 + 原子操作(2x 慢) | 多处共享所有权:缓存、观察者模式 |
| `weak_ptr` | 不拥有 | 同 `shared_ptr`(lock 时) | 打破循环引用、安全探测对象存活 |

## 一、`unique_ptr`:90% 场景的默认选择

### 三大特性

```cpp
std::unique_ptr<T> up(new T());
// 1. 独占:同一时刻只有一个 unique_ptr 拥有资源
// 2. 不可拷贝:std::unique_ptr<T> up2 = up;  // 编译错误!
// 3. 可移动:std::unique_ptr<T> up2 = std::move(up);  // OK
```

### 零开销的证据

```cpp
class Widget {
    std::unique_ptr<Impl> pImpl_;  // 成员变量
};

// unique_ptr 只有一个裸指针成员,sizeof = 8 bytes(64 位)
// 编译器生成的代码 = 裸指针的代码
// 没有引用计数、没有控制块、没有原子操作
```

### 工厂函数的标准模式

```cpp
// C++14 起 make_unique 取代 new
std::unique_ptr<Widget> create() {
    return std::make_unique<Widget>(args...);
}

auto w = create();  // 无需 new,异常安全
```

### 转让所有权

```cpp
void sink(std::unique_ptr<Widget> w) { /* 接管所有权 */ }

auto w = create();
sink(std::move(w));  // 显式转移,w 变空
// w 此时为 nullptr,不能再用
```

## 二、`shared_ptr`:共享所有权

### 实现机制

```cpp
// shared_ptr 内部有两部分:
// 1. 指向对象的指针
// 2. 指向控制块的指针(包含引用计数、weak 计数、deleter 等)
//
//   ┌─────────────┐
//   │ Object      │ ← 被管理对象
//   └─────────────┘
//         ↑
//   ┌─────────────┐
//   │ Control     │ ← 引用计数
//   │ Block       │
//   └─────────────┘
//         ↑
//   ┌─────────────┐
//   │ shared_ptr  │ ← 持有两指针
//   └─────────────┘
```

### 性能代价

- **控制块分配**:每次创建 `shared_ptr` 都要 `new` 一个控制块(用 `make_shared` 可合并)
- **原子操作**:每次拷贝/析构都要原子加减引用计数(CAS 指令,比普通读写慢 10x+)
- **缓存不友好**:多一次间接寻址(指针 → 控制块 → 对象)

### 用 `make_shared` 合并分配

```cpp
// 两次分配:对象 + 控制块
std::shared_ptr<Widget> sp1(new Widget());

// 一次分配:对象和控制块连续
auto sp2 = std::make_shared<Widget>();
// 性能更好、内存局部性更好、异常安全
```

### 共享所有权的适用场景

- **缓存**:多模块共享同一份缓存数据
- **观察者模式**:Subject 被多个 Observer 共享
- **图结构**:节点被多条边共享
- **异步回调**:回调被多个 handler 共享

**判断标准**:**能明确所有权就用 `unique_ptr`,实在需要共享再用 `shared_ptr`**。

## 三、`weak_ptr`:观察者

### 核心特性

```cpp
std::weak_ptr<Widget> wp = sp;  // 从 shared_ptr 创建

// 用之前必须 lock(),因为对象可能已死
if (auto sp2 = wp.lock()) {
    // 对象还活着,可以安全使用
    sp2->doSomething();
} else {
    // 对象已死,放弃操作
}
```

### 唯一用途:打破循环引用

```cpp
// 错误:循环引用 → 内存泄漏
struct Node {
    std::shared_ptr<Node> next;
    std::shared_ptr<Node> prev;  // 互相持有,引用计数永不为 0
};

// 正确:用 weak_ptr 打破循环
struct Node {
    std::shared_ptr<Node> next;  // 后向:共享所有权
    std::weak_ptr<Node> prev;    // 前向:不参与所有权
};
```

### 安全探测 vs 裸指针

```cpp
// 危险:对象可能已被其他 shared_ptr 释放
Widget* w = raw_ptr;       // 裸指针,可能悬空
if (w) w->doSomething();   // UB!

// 安全:lock() 返回 shared_ptr,要么有效要么空
std::weak_ptr<Widget> wp = ...;
if (auto sp = wp.lock()) {  // 原子检查并提升
    sp->doSomething();      // 安全
}
```

## 四、自定义 deleter

### unique_ptr:deleter 是类型的一部分

```cpp
// 用 FILE* + fclose 作为 deleter
std::unique_ptr<FILE, decltype(&fclose)> fp(fopen("data.txt", "r"), &fclose);
```

### shared_ptr:deleter 是控制块的一部分

```cpp
// shared_ptr 的 deleter 不影响类型,可以运行时指定
std::shared_ptr<FILE> fp(fopen("data.txt", "r"), fclose);
// 类型是 shared_ptr<FILE>,不是 shared_ptr<FILE, decltype(&fclose)>
```

这意味着 `unique_ptr` 不同 deleter 是不同类型(无运行时开销),`shared_ptr` 同一类型可接受不同 deleter(类型擦除,有轻微开销)。

## 五、`enable_shared_from_this`

### 问题:在成员函数中获取自己的 shared_ptr

```cpp
class Widget {
public:
    std::shared_ptr<Widget> getptr() {
        return std::shared_ptr<Widget>(this);  // 错误!会重复析构
    }
};
// 两次 shared_ptr 独立计数 → 析构时 double free
```

### 解决:继承 `enable_shared_from_this`

```cpp
class Widget : public std::enable_shared_from_this<Widget> {
public:
    std::shared_ptr<Widget> getptr() {
        return shared_from_this();  // 正确:返回与自己共享计数的 ptr
    }
};
```

## 六、选型决策流程

```
问题:资源该由谁拥有?
│
├─ 单一拥有者(大多数情况)
│  └─ unique_ptr(零开销)
│
├─ 多个拥有者(必须共享)
│  ├─ 出现循环引用?
│  │  └─ 改用 weak_ptr 破环(部分 weak + 部分 shared)
│  └─ 不会循环引用
│     └─ shared_ptr
│
└─ 只是观察,不拥有
   └─ weak_ptr
```

## 七、面试高频追问

**Q1: 为什么 `unique_ptr` 是零开销?**
内部只有一个裸指针(8 字节)。没有引用计数、没有控制块、没有原子操作。编译器优化后生成的机器码与裸指针完全相同(RAII 析构调用 = 直接 `delete`)。`sizeof(unique_ptr<T>) == sizeof(T*)`。

**Q2: `shared_ptr` 的性能代价在哪?**
三方面:
1. 控制块堆分配(用 `make_shared` 合并为一次)
2. 引用计数的原子操作(CAS,10x 慢于普通读写)
3. 一次额外间接寻址(指针 → 控制块 → 对象,缓存不友好)

**Q3: 循环引用怎么发生?**
A 持 `shared_ptr<B>`,B 持 `shared_ptr<A>`,两者引用计数都 ≥ 1,谁都不释放。`weak_ptr` 不增加引用计数,把其中一方改成 `weak_ptr` 即可打破。

**Q4: 能用 `unique_ptr` 代替 `shared_ptr` 吗?**
绝大多数场景可以。"能明确所有权就用 `unique_ptr`"是 Rule of Zero 时代的最佳实践。`shared_ptr` 往往是"懒得想所有权"的妥协——用多了容易引入循环引用和性能问题。**反问自己**:是真的需要共享,还是因为没想清楚所有权?

**Q5: `make_shared` 一定优于 `new` + 构造 `shared_ptr`?**
- 性能:✓ 一次分配 vs 两次,且内存局部性更好
- 异常安全:✓ 避免"new 对象 → 调函数传 shared_ptr" 中间可能抛异常导致的泄漏
- 限制:✗ 不支持自定义 deleter(因为 deleter 在构造时无法传入)
- 总结:**默认用 `make_shared`,需要自定义 deleter 才用 `new`**
