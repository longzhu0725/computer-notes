# C++20 协程

## 核心结论

协程是**能够暂停执行并在之后恢复的函数**,C++20 通过 `co_await`、`co_yield`、`co_return` 三个关键字实现。三个核心组件:**Promise 对象**控制生命周期和返回值,**Coroutine Handle** 是恢复/销毁句柄,**Awaiter** 定义 `co_await` 的等待行为。

**与线程的本质区别**:线程由 OS 内核调度,切换成本高(微秒级);协程由用户态程序自己调度,切换成本极低(纳秒级),但**不能真正并行**。

| 机制 | 调度方 | 切换开销 | 并行能力 | 适用场景 |
|------|--------|---------|---------|---------|
| 线程 | OS 内核 | 微秒级(保存寄存器+切栈) | 真并行 | CPU 密集型 |
| **协程** | 用户态程序 | 纳秒级(几个指针) | **不能并行** | I/O 密集型 |
| 回调 | — | — | 取决于外层 | 简单异步 |

## 一、协程的执行流程

### 生命周期六步骤

```
1. 分配协程帧(堆上,存局部变量/参数/恢复点)
2. 创建 Promise 对象
3. 调用 promise.get_return_object() 拿到返回给调用者的对象
4. 执行协程体,遇到 co_await/co_yield 时暂停,控制权交还调用者
5. 调用者通过 handle.resume() 恢复执行
6. 协程结束时调用 handle.destroy() 释放协程帧
```

**关键点**:协程暂停时,整个执行状态(局部变量、执行位置)都保存在协程帧里。恢复时从断点继续,不需要重新初始化。

### 一个最简单的协程

```cpp
#include <coroutine>
#include <iostream>

struct ReturnObject {
    struct promise_type {
        ReturnObject get_return_object() { return {handle_type::from_promise(*this)}; }
        std::suspend_never initial_suspend() { return {}; }  // 不挂起
        std::suspend_never final_suspend() noexcept { return {}; }  // 不挂起
        void return_void() {}
        void unhandled_exception() {}
    };
    using handle_type = std::coroutine_handle<promise_type>;
    handle_type handle;
};

ReturnObject my_coro() {
    std::cout << "step 1\n";
    co_await std::suspend_always{};  // 暂停
    std::cout << "step 2\n";
    co_return;
}

int main() {
    auto obj = my_coro();
    std::cout << "main: between\n";
    obj.handle.resume();  // 恢复执行
    obj.handle.destroy(); // 清理
}
```

输出:
```
step 1
main: between
step 2
```

## 二、三个核心组件详解

### Promise 对象

**职责**:控制协程的生命周期、返回值、异常处理。

```cpp
struct promise_type {
    // 1. 创建返回给调用者的对象
    ReturnObject get_return_object() { ... }
    
    // 2. 协程启动时是否暂停?
    std::suspend_always initial_suspend() { return {}; }  // 启动时挂起(等调用者 resume)
    std::suspend_never initial_suspend() { return {}; }   // 启动时直接执行
    
    // 3. 协程结束时是否暂停?
    std::suspend_always final_suspend() noexcept { ... }  // 结束时挂起(给调用者最后操作机会)
    
    // 4. 处理返回值
    void return_void() {}                       // 无返回值
    void return_value(T v) { value_ = v; }      // 有返回值
    
    // 5. 异常处理
    void unhandled_exception() { std::terminate(); }  // 默认终止
};
```

### Coroutine Handle

**职责**:恢复或销毁协程的句柄(类似 `std::thread` 但更轻量)。

```cpp
std::coroutine_handle<promise_type> handle;

// 恢复执行(从上次挂起点继续)
handle.resume();

// 销毁(释放协程帧)
handle.destroy();

// 检查状态
bool done = handle.done();  // 是否执行完毕

// 获取 promise
auto& p = handle.promise();
```

### Awaiter

**职责**:定义 `co_await` 表达式的等待行为。

```cpp
struct MyAwaiter {
    // 1. 是否需要挂起?(true=不挂起,直接继续;false=挂起)
    bool await_ready() { return false; }
    
    // 2. 挂起时做什么?(返回 void,或返回另一个 coroutine_handle 表示"由它来恢复我")
    void await_suspend(std::coroutine_handle<> h) {
        // 把 h 保存起来,稍后由其他线程/事件循环 resume 它
        globalQueue.push(h);
    }
    
    // 3. 恢复后返回值(作为 co_await 表达式的值)
    T await_resume() { return result_; }
};

// 使用
co_await MyAwaiter{};
```

## 三、`co_await` 的三阶段流程

```
1. await_ready() ──── true → 不挂起,直接到 step 3
                  └ false → 挂起
2. await_suspend(handle) ─── 保存 handle,挂起协程
3. await_resume() ──── 协程被 resume 后执行,返回值作为表达式结果
```

**典型应用**:异步 I/O

```cpp
// 异步读取文件
Task<std::string> readFile(const std::string& path) {
    auto handle = openAsync(path);
    co_await handle;  // 挂起,等 I/O 完成
    co_return readResult(handle);
}

// 调用方看起来像同步代码
auto content = co_await readFile("data.txt");
```

## 四、协程 vs 线程 vs 回调

### 性能对比

| 场景 | 线程切换 | 协程切换 | 差距 |
|------|---------|---------|------|
| 上下文保存 | 寄存器 + 栈(几 KB) | 几个指针(几十字节) | 100-1000x |
| 系统调用 | 需要(内核态切换) | 不需要(纯用户态) | — |
| 并行能力 | 真并行 | 协作式(单线程内) | 线程胜 |

### 代码可读性对比

**回调地狱**:
```cpp
// 三个嵌套异步操作
fetch(url1, [](data1) {
    parse(data1, [](obj1) {
        save(obj1, [](result) {
            log(result);
        });
    });
});
```

**协程线性风格**:
```cpp
Task<void> process() {
    auto data1 = co_await fetch(url1);
    auto obj1  = co_await parse(data1);
    auto result = co_await save(obj1);
    co_await log(result);
}
```

协程用**线性代码**写异步逻辑,可读性大幅提升,错误处理也简单(普通 `try-catch`)。

## 五、协程内存泄漏的避免

### 核心原则:`handle.destroy()` 必须被调用

```cpp
// 错误:忘记 destroy,协程帧永远泄漏
auto obj = my_coro();
obj.handle.resume();
// 函数结束,obj 析构,但 handle 还在 → 泄漏

// 正确:RAII 包装器
class ScopedCoroutine {
    std::coroutine_handle<> handle_;
public:
    ScopedCoroutine(std::coroutine_handle<> h) : handle_(h) {}
    ~ScopedCoroutine() {
        if (handle_) handle_.destroy();  // 析构时自动清理
    }
    void resume() { handle_.resume(); }
    // 禁止拷贝
    ScopedCoroutine(const ScopedCoroutine&) = delete;
    ScopedCoroutine& operator=(const ScopedCoroutine&) = delete;
};
```

### 避免循环引用

```cpp
// 错误:协程 A 持协程 B 的 handle,B 也持 A 的 handle → 循环引用
// 解决:用 std::weak_ptr<coroutine_handle> 持有,或明确所有权链
```

## 六、协程的适用场景

### 适合协程(强烈推荐)

- **异步 I/O**:网络请求、文件读写、数据库查询
- **生成器**:大数据集流式处理(`co_yield`)
- **状态机**:复杂状态转换用协程状态表示
- **游戏开发**:角色行为序列、AI 决策流程

### 不适合协程(用线程)

- **CPU 密集型并行计算**:矩阵乘法、图像处理——协程不能并行,纯串行
- **需要利用多核**:协程是协作式,单线程内串行;真并行必须用线程/线程池

## 七、面试高频追问

**Q1: 协程帧里存了什么?**
Promise 对象、函数参数的副本、所有局部变量、当前挂起点的位置信息。本质是**一个堆分配的结构体**,保存协程"冻结"时的全部状态。协程切换只需保存几个指针(纳秒级),而线程切换要保存整个寄存器+栈(微秒级)。

**Q2: `co_await` 的三阶段流程?**
1. `await_ready()`:是否需要挂起(true=不挂起,继续)
2. `await_suspend(handle)`:挂起时做什么(保存 handle)
3. `await_resume()`:恢复后返回值

典型应用:异步 I/O 中,`await_suspend` 把 handle 注册到事件循环,事件完成时事件循环 `resume()` 它,`await_resume()` 返回 I/O 结果。

**Q3: 怎么避免协程内存泄漏?**
核心:确保 `coroutine_handle` 一定会被 `destroy()`。实践:
1. **RAII 包装器**(析构自动 destroy)
2. `final_suspend` 返回 `suspend_always`,由持有者负责销毁
3. **避免 handle 循环引用**(用 `weak_ptr` 或明确所有权)

**Q4: 协程能替代线程吗?**
**不能**。协程解决的是"异步代码写起来难看"的问题,**不是并行计算的问题**。CPU 密集型任务还是得用线程/线程池;协程适合 I/O 密集型场景。两者是正交关系,常组合使用(线程池跑业务,协程处理 I/O 等待)。

**Q5: 协程为什么是"用户态调度"?**
线程由 OS 内核调度,涉及系统调用和内核态切换。协程**完全在用户态**——`resume()` 只是个普通函数调用,保存几个寄存器(实际上是几个指针)就完成切换。不需要陷入内核,开销极小。但代价是同一时刻只有一个协程在跑(单线程内),不能利用多核。
