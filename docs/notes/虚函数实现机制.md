# C++ 虚函数实现机制

## 核心结论

虚函数通过**虚函数表(vtable)+ 虚指针(vptr)** 实现多态。**每个含虚函数的类一张 vtable**,表里按声明顺序存各虚函数地址。**每个对象一个 vptr**,位于对象内存最前面(主流编译器),指向所属类的 vtable。`p->virtualFunc()` 在底层展开为:**取 vptr → vtable 查函数地址 → 间接调用**。这就是"动态绑定"。

| 概念 | 位置 | 内容 | 创建时机 |
|------|------|------|---------|
| vtable | 程序的**只读数据段** | 类的所有虚函数地址 | **编译期**生成 |
| vptr | 对象**内存最前** | 指向所属类 vtable | **构造时**初始化 |
| 虚函数调用 | 运行期 | 查 vtable 找地址 | 每次调用都查 |

## 一、三个核心组件

### 1. vtable:每类一张表

```cpp
class Animal {
public:
    virtual void speak() { std::cout << "..."; }
    virtual void eat() { std::cout << "eating"; }
};
class Dog : public Animal {
public:
    void speak() override { std::cout << "Woof!"; }
    // eat() 不重写,继承 Animal::eat
};
```

**编译期生成的 vtable**:

```
Animal vtable:            Dog vtable:
┌──────────────┐          ┌──────────────┐
│ 0: speak →   │──→       │ 0: speak →   │──→ Dog::speak
│   Animal::speak        │   Dog::speak
│ 1: eat →     │──→       │ 1: eat →     │──→ Animal::eat(继承)
│   Animal::eat          │   Animal::eat
└──────────────┘          └──────────────┘
```

**vtable 特点**:
- 编译期生成,存在只读数据段
- 每个类一张(不是每个对象一张)
- 表项按**声明顺序**排列
- 派生类重写时,对应位置换成自己的函数地址

### 2. vptr:每个对象一个指针

```
内存布局(主流编译器):

Dog 对象:
┌────────────────────┐
│ vptr → Dog vtable  │ ← 隐藏成员,对象最前
├────────────────────┤
│ 派生类自己的成员     │
├────────────────────┤
│ 基类成员(继承下来)  │
└────────────────────┘
```

**vptr 特点**:
- 隐藏成员,通常位于对象最前
- 构造时由编译器**自动设置**为指向当前类的 vtable
- 析构时跟随析构进度切换(构造 Derived 时,先 Base 部分 → vptr 指向 Base vtable)

### 3. 虚函数调用:动态绑定

```cpp
Animal* p = new Dog();
p->speak();  // 多态调用
```

**编译后**:
```asm
; p->speak() 展开为:
mov rax, [p]            ; 1. 取对象地址
mov rax, [rax]          ; 2. 取 vptr(对象第一个字段)
call [rax + 0]          ; 3. vtable[0] 间接调用 speak
```

**关键点**:`p` 的静态类型是 `Animal*`,但实际指向 `Dog` 对象。运行时查 vptr 找到 `Dog vtable`,`Dog vtable[0]` 是 `Dog::speak`,所以实际调 `Dog::speak`。

## 二、vptr 在对象哪里

### 主流实现

| 编译器 | vptr 位置 |
|--------|---------|
| GCC/Clang (Itanium ABI) | **对象最前** |
| MSVC | 对象最前 |
| 某些嵌入式编译器 | 末尾或不固定 |

**为什么放最前**:通过对象首地址直接拿到 vptr,不需要额外偏移。**首地址 = vptr 地址**。

### 多继承时多个 vptr

```cpp
class A { public: virtual void fa(); };
class B { public: virtual void fb(); };
class C : public A, public B {  // 多继承
public:
    void fa() override;
    void fb() override;
};
```

```
C 对象:
┌────────────────────┐
│ vptr_A → A vtable  │ ← 第一个基类的 vptr
├────────────────────┤
│ A 成员              │
├────────────────────┤
│ vptr_B → B vtable  │ ← 第二个基类的 vptr
├────────────────────┤
│ B 成员              │
├────────────────────┤
│ C 自己的成员        │
└────────────────────┘
```

通过 `A*` 调用 → 用 vptr_A,通过 `B*` 调用 → 用 vptr_B(可能涉及指针调整,见 [类型转换](/notes/类型转换.html))。

## 三、构造/析构期间的 vptr 变化

### 构造:跟随进度切换

```
new Derived() 实际执行:
1. 分配 sizeof(Derived) 内存
2. vptr 设为 Base vtable  ← 此时对象是 Base
3. 执行 Base() 构造体
4. vptr 设为 Derived vtable  ← 此时对象是 Derived
5. 执行 Derived() 构造体
```

**后果**:构造期间,虚函数被**静态绑定**到当前正在构造的类。这解释了"构造函数不能调虚函数并期望多态"——见 [this 指针](/notes/this-指针.html)。

### 析构:反过来

```
delete basePtr 实际执行:
1. 进入 Derived::~Derived(),vptr 设为 Derived vtable
2. 执行 Derived 析构体
3. 析构 Derived 成员
4. vptr 切回 Base vtable  ← 此时对象是 Base
5. 进入 Base::~Base()
6. 释放内存
```

**为什么基类析构要 `virtual`**:非虚析构时,`delete basePtr` 只调 `Base::~Base`,跳过 Derived 部分,Derived 资源泄漏。

## 四、虚函数调用的性能代价

### 一次间接寻址

```cpp
// 普通函数调用:编译期写死
mov rax, [Animal::speak 直接地址]
call rax

// 虚函数调用:运行期查表
mov rax, [p]                ; 取对象
mov rax, [rax]              ; 取 vptr
call [rax + offset]         ; vtable 查表 + 间接调用
```

**额外开销**:
- 一次指针解引用(对象 → vptr)
- 一次间接调用(vtable → 函数地址)
- 大约 1-2 条额外指令(现代 CPU 分支预测,实际差异更小)

**真正的代价**:
- **丧失内联机会**——虚函数调用编译期不知道目标,无法 inline
- 在紧密循环中是性能瓶颈
- **优化手段**:`final` 关键字(告诉编译器不会有派生类)可让编译器去虚拟化(devirtualize)

### 性能影响实测

| 场景 | 性能差异 |
|------|---------|
| 单次调用 | &lt; 1% |
| 紧密循环(10M 次) | 5-15% |
| 缓存未命中 | vptr 跨 cache line 加载,可显著变慢 |

## 五、`override` 和 `final`

### `override`:编译期检查

```cpp
class Derived : public Base {
    void f() override;  // 告诉编译器:我在重写基类虚函数
    // 编译期检查:基类有 f 吗?有 → 通过;没有 → 编译错误
};
```

**没 `override` 的话**:拼写错误(`void f()` vs `void g()`)会变成"新的虚函数",而不是重写——静默 bug。

### `final`:禁止再重写

```cpp
class Base {
public:
    virtual void f();
};
class Derived : public Base {
    void f() final;  // 禁止再重写
};
class MoreDerived : public Derived {
    void f() override;  // 编译错误:被 final 禁止
};
```

**好处**:编译器可以做去虚拟化优化(因为知道不会有更新的版本),在某些场景下让虚函数调用变成直接调用。

## 六、纯虚函数与抽象类

```cpp
class Shape {
public:
    virtual void draw() = 0;  // 纯虚函数
};
// Shape 是抽象类,不能实例化
//   Shape s;  // 错误
```

**实现**:纯虚函数的 vtable 项通常存**纯虚函数占位符**(调到 `__cxa_pure_virtual`,触发 `std::terminate`)。这是为什么"对抽象类调纯虚函数会直接 abort"。

## 七、面试高频追问

**Q1: 虚函数表(vtable)在哪?**
**编译期**生成,存在程序的**只读数据段**(.rodata 或 .data.rel.ro)。每个含虚函数的类**一张表**,不是每个对象一张。表中按**声明顺序**排列各虚函数地址。

**Q2: vptr 放在对象的什么位置?**
主流实现(GCC/Clang/MSVC)都把 vptr 放在**对象内存最前**。这样对象首地址就是 vptr 地址,无需偏移计算。但**这不是标准规定**,理论上编译器可以放任何位置(只要行为正确)。

**Q3: 多继承时 vtable 怎么处理?**
一个对象可能有**多个 vptr**,每条继承链一个。`class C : public A, public B` 时,C 对象有两个 vptr:一个指向 A vtable,一个指向 B vtable。通过不同基类指针调虚函数,走的是不同的 vptr。

**Q4: 虚函数调用比普通函数慢多少?**
多一次指针解引用 + 一次间接调用,通常 1-2 条指令的开销。现代 CPU 有分支预测和缓存,实际差异 &lt; 5%。**真正的代价**是**丧失内联机会**——虚函数无法 inline,紧密循环中是瓶颈。`final` 关键字可以让编译器去虚拟化,恢复内联。

**Q5: 为什么基类析构要 `virtual`?**
非虚析构时,`delete basePtr` 只会调 `Base::~Base`,跳过派生类析构,派生类申请的内存/资源泄漏。**virtual 析构**让析构走 vtable 查表,能正确调到 `Derived::~Derived` → 清理 → 然后调 `Base::~Base`。**有继承且可能通过基类指针删除,就 virtual**。
