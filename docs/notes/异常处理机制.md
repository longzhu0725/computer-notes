# C++ 异常处理机制

## 核心结论

C++ 异常处理基于 `try-throw-catch` 三件套,核心机制是**栈展开**(Stack Unwinding):从抛出点沿调用链向上逐层退出函数,途中自动调用所有局部对象的析构函数。配合 [RAII 机制](/notes/raii-机制.html),即使发生异常资源也不会泄漏。`noexcept` 标记的函数如果实际抛异常,程序直接 `terminate`,不做栈展开。

| 语法元素 | 作用 | 关键约束 |
|---------|------|---------|
| `try` | 定义监控范围 | 标记"这段代码可能出错" |
| `throw` | 抛出异常对象 | 立即终止当前函数执行 |
| `catch` | 按类型匹配捕获 | 派生类必须写在基类前面 |
| `catch(...)` | 兜底捕获所有异常 | 必须放在最后 |
| `noexcept` | 声明函数不抛异常 | 违反则直接 `terminate` |

## 异常执行流程

当 `throw` 被执行时,程序不会继续往下走,而是立即开始**栈展开**——从当前函数退出,沿着调用链一层一层往上找,直到找到一个类型匹配的 `catch` 块。这个过程中,每退出一层函数,该层所有局部对象的析构函数都会被自动调用。

这就是异常处理的精髓:**错误处理和资源清理解耦**。你不需要在每一层手动检查错误码、手动释放资源——[RAII 对象](/notes/raii-机制.html)的析构函数会自动搞定。

## 异常捕获的规则

- `catch` 块按顺序匹配,找到第一个类型兼容的就进入
- 派生类异常的 `catch` 要写在基类前面,否则永远匹配不到
- `catch(...)` 能捕获所有异常,通常放在最后做兜底
- 异常对象通常用 `const T&` 捕获(避免切片 + 零拷贝)

### 切片问题示例

```cpp
try {
    throw DerivedException("error");
} catch (const BaseException& e) {  // 正确:保留多态
    std::cout << e.what();
} catch (const std::exception& e) {  // 错误:放在基类前面永远匹配不到
    // ...
}
```

## noexcept 的作用

C++11 引入 `noexcept` 说明符,告诉编译器"这个函数保证不抛异常"。编译器可以据此做优化:

- **STL 容器扩容**:`vector::push_back` 时,如果移动构造是 `noexcept` 的,就用移动而不是拷贝(否则用拷贝保证异常安全)
- **析构函数**:C++11 后**隐式 noexcept**,不能抛异常
- **优化代码生成**:编译器可以省略异常处理的栈表(entry table),代码更小更快

如果标了 `noexcept` 的函数实际抛了异常,程序直接 `std::terminate()`,**不做栈展开**——所以不要在析构中抛异常。

## 异常安全的三个级别

| 级别 | 含义 | 典型实现 |
|------|------|---------|
| 基本保证 | 异常发生后,对象处于合法状态,资源不泄漏 | RAII 包装所有资源 |
| 强保证 | 异常发生后,程序状态回滚到操作之前 | copy-and-swap 模式 |
| 不抛保证 | 函数保证不抛异常 | `noexcept` 标记 + 内部吞掉异常 |

## 异常 vs 错误码

| 维度 | 异常 | 错误码 |
|------|------|--------|
| 适用场景 | 低频、不可恢复错误 | 高频、可预期错误 |
| 性能开销 | 抛出时有栈展开代价 | 每次调用都要检查 |
| 可忽略性 | 不会被忽略(栈展开强制处理) | 容易被忽略 |
| 构造失败 | 唯一选择 | 无法表达 |
| 跨语言 | 不能跨越 C ABI | 可以 |

实际项目通常**混用**:底层库用错误码(性能、可控),上层业务逻辑用异常(代码清晰、不可忽略)。构造函数只能用异常报告失败,这是没得选的。

## 析构函数为什么不能抛异常

栈展开过程中会调用局部对象的析构函数。如果这时候析构函数又抛了异常,就出现了"异常嵌套"——C++ 标准规定这种情况直接调用 `terminate` 终止程序。

正确做法:析构函数必须标 `noexcept`(C++11 隐式 noexcept),内部的异常必须自己吞掉:

```cpp
~FileHandle() noexcept {
    try {
        if (fp_) std::fclose(fp_);  // 失败只能记日志
    } catch (...) {
        log("close failed");
        // 不能 rethrow
    }
}
```

## 性能开销

现代编译器用"**零开销异常**"(zero-cost exception)实现:

- 正常执行路径**没有额外开销**(异常表存在 .eh_frame 段,不进代码缓存)
- 只有真正抛出异常时才付出代价(栈展开、查找 catch 块)

所以异常适合处理"异常情况":如果某个错误发生的概率很高(比如每次循环都可能触发),用错误码更合适;如果是真正的异常情况(比如内存耗尽),用异常没问题。

## 面试高频追问

**Q1: 为什么必须按引用捕获异常?**
按值捕获会发生**对象切片**——如果抛出的是派生类异常,按基类值捕获会丢失派生类信息,多态失效。按引用捕获保持多态性,而且避免了不必要的拷贝。用 `const` 引用更好,因为你通常不需要修改异常对象。

**Q2: 构造函数抛异常会怎样?**
- 对象没有构造完成,析构函数不会被调用
- 但 C++ 保证:已经构造完成的成员变量会被正确析构(成员按声明逆序析构)
- 所以 RAII 类的成员也必须是 RAII 类型(unique_ptr 优于裸指针)

**Q3: throw; 后面不跟表达式是什么意思?**
**重新抛出**当前正在处理的异常,只能在 catch 块中使用。常见于包装异常:

```cpp
try { /* ... */ }
catch (const std::exception& e) {
    log("operation failed: " + std::string(e.what()));
    throw;  // 重新抛给上层
}
```
