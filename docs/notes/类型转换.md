# C++ 四种类型转换

## 核心结论

C++ 用四种**命名转换**(`static_cast`、`dynamic_cast`、`const_cast`、`reinterpret_cast`)替代 C 风格的 `(type)value`,核心目的是**让转换意图显式化**——编译器能帮你检查错误,代码读者能立刻看出"在做什么转换"。

| 转换类型 | 检查时机 | 运行时开销 | 安全性 | 典型用途 |
|---------|---------|-----------|--------|---------|
| `static_cast` | 编译期 | 零 | 中(信任程序员) | 基础类型、继承向上/向下 |
| `dynamic_cast` | 运行期(RTTI) | 有 | 高(失败可检测) | 多态类型的向下转换 |
| `const_cast` | 编译期 | 零 | 低(可改 const) | 加/去 const/volatile |
| `reinterpret_cast` | 编译期 | 零 | 极低(比特位重解释) | 指针/整数互转、底层 hack |

## 一、为什么 C++ 要废弃 C 风格转换

C 风格 `(type)value` 是"瑞士军刀"——它能做四种转换中的任何一种,代码读者和编译器都无法判断"到底在干什么"。

```cpp
// C 风格:看不出意图
(void*)ptr;            // 是 const_cast?reinterpret_cast?static_cast?
(int)x;                // 是数值截断?reinterpret_cast?
Base* b = (Base*)d;    // 是 static_cast 还是 reinterpret_cast?
```

四种命名转换是"四把专用工具"——看代码就知道在做什么:

```cpp
// C++ 风格:意图明确
static_cast<int>(x);                       // 数值截断,安全
static_cast<Base*>(d);                     // 静态断言:我知道 d 真的指向 Base
dynamic_cast<Base*>(d);                    // 运行时检查,失败返回 nullptr
const_cast<int*>(p);                       // 我确定底层不是 const,要改
reinterpret_cast<uintptr_t>(p);            // 拿地址当整数,无安全保证
```

## 二、`static_cast`:用途最广的编译期转换

### 能做的事

```cpp
// 1. 基础类型之间(隐式转换的显式版本)
int i = static_cast<int>(3.14);          // 3,截断
double d = static_cast<double>(i);       // 3.0

// 2. void* 转具体指针(与 C 风格不同,需显式)
int* p = static_cast<int*>(malloc(sizeof(int)));

// 3. 继承向上转换(安全,等价于隐式转换)
Derived* d = new Derived();
Base* b = static_cast<Base*>(d);         // 总是成功

// 4. 继承向下转换(不安全,不检查)
Base* b = getFromSomewhere();
Derived* d = static_cast<Derived*>(b);   // 如果 b 实际不是 Derived,UB
```

### 关键特性

- **编译期完成**,零运行时开销
- 编译器会做有限的检查(类型必须"逻辑相关"——基础类型之间、有继承关系的指针/引用、void* 转具体类型)
- 向下转换时**不**检查对象实际类型——转错了直接 UB(访问错误内存、虚表错乱、崩溃)

## 三、`dynamic_cast`:运行期类型检查的向下转换

### 工作原理

`dynamic_cast` 通过 RTTI(Run-Time Type Information,运行时类型信息)检查对象的实际类型。每个有虚函数的类,C++ 编译器会生成一个 `type_info` 对象,`dynamic_cast` 通过比较 `type_info` 来判断转换是否合法。

```cpp
// 多态基类
class Shape { virtual ~Shape() = default; };
class Circle : public Shape { public: void draw(); };
class Square : public Shape { public: void draw(); };

// 安全的向下转换
Shape* s = getShape();  // 实际可能是 Circle 或 Square
if (Circle* c = dynamic_cast<Circle*>(s)) {
    c->draw();          // 确实是 Circle,放心用
} else if (Square* sq = dynamic_cast<Square*>(s)) {
    sq->draw();
}
```

### 失败处理

```cpp
// 指针转换:失败返回 nullptr
Circle* c = dynamic_cast<Circle*>(s);
if (!c) { /* 不是 Circle */ }

// 引用转换:失败抛 std::bad_cast
try {
    Circle& c = dynamic_cast<Circle&>(*s);
} catch (const std::bad_cast& e) {
    // 处理
}
```

### 性能代价

`dynamic_cast` 需要:
1. 对象有虚函数(否则编译失败)
2. 遍历继承链或查 vtable,比较 `type_info`
3. 跨 DLL 边界时可能更慢(不同 DLL 的 `type_info` 不通用)

对于性能敏感的代码(游戏循环、高频交易),频繁 `dynamic_cast` 是优化热点。

## 四、`const_cast`:只改 const/volatile

### 唯一合法用途

```cpp
// 场景:传 const 给老 API(老 API 参数非 const)
// 错误做法:复制(浪费内存)
void legacyProcess(char* str);
const std::string& s = getString();
legacyProcess(const_cast<char*>(s.data()));  // OK,因为 s 本身非 const
```

注意:**只有当原对象本身不是 const 时**,`const_cast` 改后修改才安全。

```cpp
// 安全:原对象非 const
int x = 42;
const int& cr = x;
int& r = const_cast<int&>(cr);
r = 100;  // OK,合法修改

// 不安全:原对象就是 const(UB)
const int y = 42;
const int& cy = y;
int& ry = const_cast<int&>(cy);
ry = 100;  // UB! 编译器可能把 y 优化为常量,改的是栈上副本
```

### 易混淆点

```cpp
// const_cast 只能改"顶层 const"(指针本身是不是 const)
// 它能"穿透"间接层去掉"底层 const"
const int* p = &x;             // 底层 const(指向的东西是 const)
int* q = const_cast<int*>(p);  // 合法,只是去掉了"不能通过 q 改 x"的承诺
```

## 五、`reinterpret_cast`:比特位重解释

### 本质

**不做任何转换计算**,纯粹把一块内存的比特位"当另一种类型来读"。

```cpp
// 1. 指针转整数(拿地址)
int* p = &x;
uintptr_t addr = reinterpret_cast<uintptr_t>(p);

// 2. 整数转指针(从地址重建指针)
int* p2 = reinterpret_cast<int*>(addr);

// 3. 不相关指针类型互转
struct A { int a; };
struct B { int b; };
A* pa = new A();
B* pb = reinterpret_cast<B*>(pa);  // 不保证 pb 指向有效 B 对象
```

### 适用场景(99% 的应用开发用不到)

- 驱动开发(把设备寄存器地址当指针)
- 序列化/反序列化(自定义二进制协议)
- 内存池实现(从 `void*` 还原原始类型)
- 哈希/指纹计算(把对象内存当 `char[]` 读取)
- 与 C ABI 交互(Windows COM、D-Bus、FFI)

**原则**:能用 `static_cast` 就别用 `reinterpret_cast`。`reinterpret_cast` 不保证结果有效,用了就要自己负责。

## 六、四种转换的本质差异

### 在多重继承中调整指针

```cpp
struct Base1 { int x; };
struct Base2 { int y; };
struct Derived : Base1, Base2 {};
Derived d;
Base2* b2 = &d;   // 指针值会调整(sizeof(Base1) 字节)
```

- `static_cast<Base2*>(&d)`:**会**调整指针值,得到正确的 Base2 子对象地址
- `reinterpret_cast<Base2*>(&d)`:**不**调整,直接把 `&d` 当 Base2 指针用,得到错误地址

这就是"指针值调整"——`static_cast` 知道类型关系,会做算术调整;`reinterpret_cast` 不知道,只做比特位重解释。

## 七、面试高频追问

**Q1: static_cast 和 dynamic_cast 向下转换的区别?**
- `static_cast`:编译期完成,不检查,转错直接 UB
- `dynamic_cast`:运行期 RTTI 检查,失败返回 `nullptr`(指针)或抛 `std::bad_cast`(引用)
- 代价:`dynamic_cast` 有性能开销(查 vtable/type_info)

**Q2: const_cast 什么时候安全?**
只有"原对象本身不是 const"时安全。`int x = 42; const int& cr = x; int& r = const_cast<int&>(cr); r = 100;` 合法,因为 `x` 本身是 `int`。`const int y = 42; int& r = const_cast<int&>(y);` 是 UB,编译器可能把 `y` 优化为常量,你改了栈上副本。

**Q3: reinterpret_cast 和 static_cast 处理指针转换的区别?**
- `static_cast` 在多重继承中**会**调整指针值(因为 Base2 子对象地址可能和 Derived 地址不同)
- `reinterpret_cast` **不**调整,纯粹把比特位原样当新类型用
- 比喻:`static_cast` 是"知道类型关系的安全转换",`reinterpret_cast` 是"我不在乎类型,只要比特位"

**Q4: 为什么 C++ 不允许 C 风格转换用于类层次?**
C 风格 `(Base*)derived_ptr` 在多重继承、虚拟继承下行为未定义——是否调整指针值、是否查 vtable 由编译器决定。C++ 的 `static_cast` 明确定义了行为:`static_cast<Base*>(Derived*)` 一定调整指针。这是为了让多态代码在所有平台/编译器下行为一致。

**Q5: dynamic_cast 为什么要求基类有虚函数?**
`dynamic_cast` 通过 RTTI 工作,RTTI 依赖 vtable(虚函数表)。无虚函数的类没有 vtable,RTTI 不存在,`dynamic_cast` 编译失败。这条规则是"防止意外成本"——你不该为没有多态的类付出 RTTI 代价。
