# 内存泄漏、野指针和内存越界

## 核心结论

三大内存陷阱的本质区别:

| 问题 | 核心机制 | 现象 | 主要风险 |
|------|---------|------|---------|
| 内存泄漏 | 申请后丢失指针 | 内存被白白占着 | 长期运行 OOM |
| 野指针 | 访问已释放的内存 | 行为未定义 | 看起来"正常"最危险 |
| 内存越界 | 访问合法范围外 | 覆盖其他数据 | 安全漏洞根源 |

C++ 里最有效的统一解决方案是 [RAII 机制](/notes/raii-机制.html):用智能指针(`unique_ptr`/`shared_ptr`)管理堆内存,离开作用域自动释放。

## 内存泄漏

程序通过 `new`/`malloc` 在堆上申请了内存,但后来指针被覆盖、函数返回、或者异常跳出,导致这块内存的地址丢了,再也没法释放。短期看程序不会崩,但长期运行(服务器、嵌入式设备)内存会被慢慢吃光,最终分配失败或系统 OOM 杀进程。

### 典型泄漏场景

```cpp
void leak_demo() {
    int* p = new int[100];
    p = nullptr;       // 地址丢失,内存泄漏
    return;
}
```

```cpp
void leak_exception() {
    auto* p = new int[100];
    do_something();    // 抛异常
    delete[] p;        // 永远到不了
}
```

### 避免方法

1. **首选 RAII**:`std::unique_ptr<int[]> p = std::make_unique<int[]>(100);`
2. **避免裸 new**:用 `make_unique` / `make_shared`(同时防止异常泄漏)
3. **检测工具**:Valgrind、AddressSanitizer、Visual Studio CRT
4. **代码规范**:每条路径都要有对应的释放,异常路径通过 RAII 自动覆盖

## 野指针

指针本身还存在,但它指向的内存已经被 `delete`/`free` 了。这时候通过这个指针读写,行为是**未定义的**——可能读到旧数据(看起来"正常"),可能读到垃圾值,也可能直接 SIGSEGV 崩溃。

**最危险的是"看起来正常"的情况**,bug 潜伏很久才爆发。

### 野指针产生途径

```cpp
// 1. delete 后没置空
int* p = new int(42);
delete p;
*p = 10;  // 野指针写入,UB

// 2. 返回局部变量地址
int* bad() {
    int x = 5;
    return &x;  // x 在函数返回时销毁
}

// 3. 指针指向已移动的对象
std::unique_ptr<int> p1 = std::make_unique<int>(1);
std::unique_ptr<int> p2 = std::move(p1);
*p1 = 2;  // p1 已经为空,UB
```

### 避免方法

1. **`delete` 后立刻置 `nullptr`**:`delete p; p = nullptr;`
2. **不要返回局部变量的地址**
3. **用智能指针**:析构后自动失效,访问会抛 `std::bad_weak_ptr` 或返回空指针
4. **工具检测**:AddressSanitizer 能精准捕获 use-after-free

## 内存越界

访问数组时下标超出 `[0, size-1]` 的范围,或者往 `malloc(10)` 分配的 10 字节之外写数据。越界写入可能覆盖相邻变量、堆管理元数据、甚至函数返回地址,是**缓冲区溢出攻击**的根源。

### 越界的隐藏危害

```cpp
int arr[10] = {};
arr[10] = 42;     // 越界 1 字节,可能踩到其他栈变量
arr[-1] = 42;     // 越界反方向,可能踩到 rbp/saved registers

char* buf = (char*)malloc(10);
strcpy(buf, "0123456789");  // 写 11 字节,溢出 1 字节,可能覆盖堆元数据
```

### 避免方法

1. **用 `std::vector` 代替裸数组**:debug 模式下有边界检查(`at()` 抛异常)
2. **ASan**:编译时开 `-fsanitize=address` 检测越界,性能损耗 ~2x
3. **下标检查**:循环中严格检查边界
4. **字符串安全函数**:`strncpy` / `snprintf` 替代 `strcpy` / `sprintf`
5. **C++20 `std::span`**:明确长度信息,避免裸指针丢失大小

## 智能指针的局限

**Q: 智能指针能完全防止内存泄漏吗?**

大多数情况能,但 `shared_ptr` **循环引用**会导致引用计数永远不归零,内存永远不释放:

```cpp
struct Node {
    std::shared_ptr<Node> next;
    std::shared_ptr<Node> prev;  // 循环引用!
};
auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;
b->prev = a;  // a 和 b 引用计数都是 2,无法释放
```

**解决办法**:用 `std::weak_ptr` 打断引用环——`prev` 用 `weak_ptr`,不增加引用计数。

另外智能指针管不了非内存资源(文件句柄、socket),那些需要自定义 deleter 或者用 [RAII 封装类](/notes/raii-机制.html)。

## 野指针为什么有时候"能用"

`free`/`delete` 只是告诉分配器这块内存可以回收,**并不会立刻清零或归还操作系统**。短期内数据还在原地,所以野指针读到的可能是旧值——这恰恰是最危险的,因为 bug 不会立刻暴露,等内存被重新分配给别人才会出问题。

```cpp
int* p = new int(42);
delete p;  // 内存归还分配器,数据未清零
std::cout << *p;  // 大概率打印 42,看起来"正常"
*p = 100;  // 大概率"成功",但已经踩别人的内存
```

## 越界与堆溢出的关系

| 越界位置 | 名称 | 攻击面 |
|---------|------|--------|
| 栈缓冲区 | 栈缓冲区溢出 | 覆盖返回地址劫持控制流 |
| 堆缓冲区 | 堆缓冲区溢出 | 覆盖堆元数据或相邻对象的 vtable |
| 全局缓冲区 | BSS/data 段溢出 | 覆盖其他全局变量或函数指针 |

三者都是越界写入,只是攻击面不同。栈溢出可以 RCE(远程代码执行),堆溢出可以控制流劫持,都是高危漏洞的入口。

## 泄漏为什么不会立刻崩

因为泄漏只是内存没释放,程序本身的逻辑没有错误。**操作系统的虚拟内存机制**让进程可以持续申请,直到地址空间或物理内存耗尽。

| 运行场景 | 泄漏敏感度 | OOM 时间 |
|---------|-----------|---------|
| 桌面 GUI | 低 | 几天-几周 |
| Web 服务器 | 高 | 几小时-几天 |
| 嵌入式/IoT | 极高 | 几分钟-几小时 |
| 高频交易 | 极高 | 几秒钟 |

长期运行的服务对泄漏特别敏感,所以服务器端 C++ 代码**严禁裸 new**——这是行业铁律。

## 调试工具对比

| 工具 | 检测泄漏 | 检测野指针 | 检测越界 | 性能损耗 |
|------|---------|-----------|---------|---------|
| Valgrind | ✅ | ✅ | ✅ | 10-50x |
| AddressSanitizer | ✅ | ✅ | ✅ | 2x |

## 面试高频追问

**Q1: `new[]` 配 `delete` 还是 `delete[]`?**
必须配 `delete[]`!否则只析构一个元素,其他元素的析构函数不被调用,造成内存泄漏(如果元素有非平凡析构)。`new` 配 `delete`,`new[]` 配 `delete[]`,**类型必须严格匹配**。

**Q2: `malloc` 配 `delete` 行不行?**
**不行**。`malloc`/`free` 和 `new`/`delete` 是两套独立的内存管理机制(虽然 `new` 底层通常调 `malloc`)。混用会导致未定义行为:不同分配器的 metadata 不兼容,析构可能根本不被调用。

**Q3: 如何排查"偶发性崩溃"?**
大概率是野指针或越界。开启 ASan 重跑,通常能立刻定位到崩溃点。如果 ASan 跑不出来,问题可能是线程安全——加 ThreadSanitizer 看看。
