# C++ 堆与栈的区别

## 核心结论

栈是**编译器自动管理**的高效内存,适合生命周期短、大小固定的局部变量;堆是**程序员手动管理**的大容量内存,适合生命周期长、大小动态的对象。栈分配仅需移动栈指针(一条 CPU 指令),堆分配需遍历空闲链表,可能触发系统调用,慢几个数量级。

| 维度 | 栈(Stack) | 堆(Heap) |
|------|-----------|----------|
| 管理方 | 编译器自动 | 程序员手动(`new`/`delete`) |
| 大小 | 1-8 MB(Linux 8M,Win 1M) | 几 GB(32 位系统理论 2-3 GB) |
| 分配速度 | **O(1) 一条指令** | O(n) 查找空闲块,可能系统调用 |
| 生长方向 | 向低地址 | 向高地址 |
| 碎片 | **无**(后进先出) | 有(中间释放的小块难复用) |
| 线程安全 | **是**(每线程独立栈) | 否(共享堆,需加锁) |
| 溢出原因 | 递归太深/局部数组过大 | 频繁分配不释放 |
| 释放时机 | 作用域结束自动 | 手动 `delete` |

## 一、进程内存布局(从低到高)

```
低地址
┌──────────────────┐
│   代码段 (.text)  │  ← 机器指令,只读
├──────────────────┤
│   数据段 (.data)  │  ← 已初始化的全局/静态变量
├──────────────────┤
│   BSS 段          │  ← 未初始化的全局/静态(自动清零)
├──────────────────┤
│   堆(向上生长)    │  ← malloc/new 分配
│        ↑          │
│   可用空间         │
│        ↓          │
│   栈(向下生长)    │  ← 局部变量/函数参数
├──────────────────┤
│   内核空间         │  ← OS 占用
└──────────────────┘
高地址
```

**关键**:堆和栈**相向而行**,中间是可用空间。堆满会向高地址扩展,栈满会向低地址扩展。极端情况:递归太深 + 大量堆分配 → 两边相遇 → 程序崩溃。

## 二、栈的细节

### 栈帧(Stack Frame)

```cpp
void f() {
    int a = 1;           // 局部变量 → 栈帧
    int arr[100];        // 局部数组 → 栈帧
    // 调用 g() 时,压入新栈帧
}
void g() {
    int b = 2;           // 独立的栈帧
    // g 返回时,栈帧弹栈,b 自动销毁
}
```

每次函数调用,编译器在栈上分配一帧(局部变量、参数、返回地址),函数返回自动弹栈销毁。

### 栈溢出的原因

```cpp
// 1. 递归太深
void recursive() {
    char buf[1024];      // 每层 1KB
    recursive();          // 无限递归
}
// 第 8192 次调用时(8MB 栈)就溢出

// 2. 局部数组过大
void big() {
    int arr[1024 * 1024 * 4];  // 16MB 数组,直接溢出
}

// 3. 栈帧过大(大量局部变量)
void many() {
    int a, b, c, ..., zzz[10000];  // 几 MB 局部变量
}
```

**解法**:递归改迭代、大数组放堆(`vector` 代替原生数组)、拆函数减少每帧占用。

### 栈的线程安全

```cpp
// 每个线程有自己独立的栈
void worker() {
    int local = 42;     // 线程 worker 的栈上
}
void main() {
    std::thread t(worker);
    int main_local = 1; // 线程 main 的栈上
    t.join();
}
// worker 看到的 local 和 main 看到的 main_local 不在同一块内存
```

**这意味着**:**局部变量天然线程安全**(没共享就没竞争),只用关心堆上的共享数据。

## 三、堆的细节

### 堆分配的完整路径

```cpp
int* p = new int(42);

// 1. 编译器算出 sizeof(int) = 4
// 2. 调用 operator new(4)
// 3. operator new 内部调 malloc(4)
// 4. malloc 在空闲链表查找 4 字节块
// 5. 没找到 → mmap/brk 系统调用向内核申请新页面
// 6. 返回指针
// 7. 在指针上调用 int 的"构造函数"(基本类型是零初始化)
```

### 堆碎片

```cpp
// 假设堆上有连续 1MB 空间
char* a = new char[512 * 1024];   // 占前 512K
char* b = new char[256 * 1024];   // 占接下来 256K
char* c = new char[128 * 1024];   // 占接下来 128K
delete[] b;                        // 释放中间 256K
// 此时 1MB 里剩 512K(前)+ 256K(空)+ 128K(后)
// 申请 200K?可以(放中间 256K)
// 申请 300K?不行!中间最大连续只有 256K
// → 外部碎片:总空间够,但分散无法满足大请求
```

### 多线程的锁竞争

```cpp
// 堆是所有线程共享的
void worker() {
    int* p = new int(42);  // 需要加锁保护 malloc 内部数据结构
}
// N 个线程并发 new,锁竞争严重 → 性能下降
```

**优化**:
1. `tcmalloc` / `jemalloc` 替代 `glibc malloc`,减少锁竞争
2. 线程本地存储(TLS)池,各线程独立分配
3. 内存池预分配

## 四、对比总结表

| 维度 | 栈 | 堆 |
|------|----|----|
| 分配速度 | **1 条 CPU 指令** | 系统调用 1000+ ns |
| 典型容量 | 1-8 MB | 数 GB |
| 碎片 | **无** | 严重(高频分配场景) |
| 线程安全 | **天然**(每线程独立) | 否(需加锁) |
| 生命周期 | 作用域结束自动 | 手动控制 |
| 地址稳定性 | 差(函数返回失效) | 好(整个生命周期稳定) |
| 越界后果 | **立即崩溃**(栈溢出) | 静默破坏相邻堆块 |
| 适用对象 | 局部变量、临时对象 | 跨函数对象、大对象、动态大小 |

## 五、地址稳定性的实战影响

### 栈地址不能跨函数

```cpp
int* bad() {
    int x = 42;
    return &x;  // 危险:x 在函数结束时销毁
}
// 调用方拿到的指针指向已释放的栈内存
int* p = bad();
*p;  // UB!栈内存已被覆盖
```

### 堆地址可以跨函数

```cpp
int* good() {
    int* p = new int(42);  // p 指向堆
    return p;               // OK,堆对象不会自动销毁
}
// 调用方负责 delete
int* p = good();
*p;        // 42
delete p;  // 必须手动释放
```

**这条规则决定了**:
- 返回对象指针 → 用 `new` 或 `make_unique`
- 返回对象值 → 用 RAII(`std::vector`、`std::string`)
- **永远不要返回局部变量的指针/引用**

## 六、什么时候用哪个

### 用栈

- ✓ 局部变量(几乎所有)
- ✓ 小型对象(&lt; 几 MB)
- ✓ 短生命周期(作用域内)
- ✓ 临时对象(函数参数、返回值)
- ✓ 状态机、有限数据结构

### 用堆

- ✓ 大对象(&gt; 几 MB)
- ✓ 长生命周期(跨多个函数)
- ✓ 动态大小(运行时才知道)
- ✓ 多线程共享
- ✓ 容器底层(`vector` 内部用堆)

### 现代 C++ 的最佳实践

```cpp
// 1. 默认用栈上对象
int x = 42;
Widget w;  // 栈上

// 2. 需要动态大小 → 用容器
std::vector<int> v(1000);  // 内部堆,外部栈,自动管理

// 3. 需要共享所有权 → 用智能指针
auto p = std::make_unique<Widget>();
auto sp = std::make_shared<Widget>();

// 4. 性能敏感 → 内存池
// 一次预分配大块堆,内部细粒度管理

// 5. 避免:裸 new/delete
// 这是 C++ 内存 bug 的最大来源
```

## 七、面试高频追问

**Q1: 栈溢出是怎么发生的?**
**三类原因**:
1. 递归太深(每层 1KB,8MB 栈 8000 层就溢出)
2. 局部数组过大(几十万字节级)
3. 栈帧过大(大量局部变量)
**OS 收到 SIGSEGV,终止进程**。修复:递归改迭代、大数组放堆(用 `vector`)、拆函数。

**Q2: 堆分配为什么比栈慢?**
**三个原因**:
1. **查找开销**——空闲链表搜索,O(n) 复杂度
2. **锁竞争**——多线程共享堆数据结构,加锁解锁
3. **系统调用**——空间不够时 `brk`/`mmap` 进内核(微秒级)
栈分配只需修改 `RSP` 寄存器,一条指令完成。

**Q3: 实际项目怎么管理堆内存?**
**现代 C++ 原则**:**尽量不手动 `new`/`delete`**。
- 智能指针(`unique_ptr` / `shared_ptr`)自动管理生命周期
- 容器(`vector`、`string`)代替手写数组
- 遵循 RAII:构造时获取,析构时释放
- 性能敏感(游戏、高频交易)用**内存池**预分配

**Q4: 堆对象和栈对象的行为差异?**
| 维度 | 栈 | 堆 |
|------|----|----|
| 生命周期 | 作用域结束**自动** | 手动 `delete` |
| 地址稳定性 | **差**(返回失效) | 好(整个生命周期) |
| 拷贝 | 值拷贝(直接) | 通过指针(避免拷贝) |
| 适合 | 局部/短生命周期 | 跨函数/共享 |

**Q5: 栈为什么不会碎片化?**
栈是**严格后进先出**——后分配的先释放,内存块总是连续归还。堆是任意顺序申请/释放,中间释放的小块成为"洞",无法被大块请求利用。栈的 LIFO 特性让它天然避免碎片。
