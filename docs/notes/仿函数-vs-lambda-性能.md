# 仿函数 vs Lambda 性能

## 核心结论

**性能基本相当**。因为编译器会把 lambda 表达式转换成一个**匿名的仿函数类**——有 `operator()` 的结构体。捕获的变量变成这个结构体的成员。所以从编译器的视角看,lambda 和手写仿函数没有本质区别,都能被内联优化。

在 `-O2`/`-O3` 优化级别下,两者的性能差异通常**小于 1%**,可以忽略。

| 维度 | 仿函数(Functor) | Lambda 表达式 |
|------|----------------|--------------|
| 本质 | 有 `operator()` 的类/结构体 | 编译器生成的匿名类(等价于仿函数) |
| 性能 | 优(可内联) | **优(可内联)** |
| 性能差异 | &lt; 1%(`-O2` 下) | &lt; 1%(`-O2` 下) |
| 命名 | 有类名,易于调试 | 匿名(编译器内部命名) |
| 跨编译单元复用 | ✅ | ❌(局部) |
| 继承/多态 | ✅ | ❌ |
| 就地定义 | ❌(需单独写类) | ✅ |
| 状态管理 | 显式成员 | 隐式(capture) |

## Lambda 的编译器转换

当你写 `[x](int a) { return a * x; }` 时,编译器实际生成的是:

```cpp
// 编译器内部(等价的)
struct __lambda_x_capture_42 {
    int x;  // 捕获的变量
    int operator()(int a) const {  // 默认 const
        return a * x;
    }
};
```

然后用捕获的 `x` 初始化这个结构体。所以 **lambda 和仿函数在底层是同一个东西**——编译器对它们做的优化(内联、常量传播等)完全一样。

**关键点**:`operator()` 默认为 `const`,捕获的变量是 const 成员。如果需要修改捕获,lambda 要 `mutable`:

```cpp
int n = 0;
auto f = [n]() mutable { ++n; };  // mutable 才能修改 n(但 n 是副本)
```

## 真正影响性能的因素

不是"用 lambda 还是仿函数",而是:

| 因素 | 影响 | 建议 |
|------|------|------|
| 捕获方式 | 值捕获会拷贝(大对象开销大) | 大对象用 `[x = std::move(obj)]` 移动捕获 |
| 引用捕获 | 零拷贝但有悬垂风险 | 确认 lambda 不比被捕获变量活得久 |
| **`std::function` 包装** | 引入虚函数 + 可能堆分配 | **热路径不要用** |
| 模板 vs 类型擦除 | 模板可内联,function 不能 | STL 算法直接接收 lambda,不要包成 function |

### std::function 为什么慢?

`std::function` 是**类型擦除**的通用可调用对象容器:

- 内部用**虚函数**或**函数指针**实现多态调用
- 编译器**无法内联**
- 如果 lambda 捕获的数据超过 16~32 字节,`std::function` 会在**堆上分配**内存

**所以在性能敏感的热路径上,不要用 `std::function` 传递 lambda**——直接用模板参数接收:

```cpp
// 错误(慢)
void process(std::function<int(int)> f, int x) {
    return f(x);
}

// 正确(快,可内联)
template <typename F>
auto process(F&& f, int x) {
    return std::forward<F>(f)(x);
}
```

## 捕获方式对性能的影响

| 捕获方式 | 语法 | 开销 | 风险 |
|---------|------|------|------|
| 值捕获 | `[x]` | 拷贝(大对象代价大) | 无 |
| 引用捕获 | `[&x]` | 零拷贝(只存引用) | **悬垂引用** |
| 移动捕获(C++14) | `[x = std::move(obj)]` | 移动(无拷贝) | 无 |
| 隐式值捕获 | `[=]` | 全部按值拷贝 | 大对象意外拷贝 |
| 隐式引用捕获 | `[&]` | 零拷贝 | 全部悬垂风险 |

```cpp
// 错误:意外大拷贝
std::vector<int> big(1000000);
auto f = [=]() { return big.size(); };  // 拷贝整个 vector!

// 正确:显式移动或引用
auto f = [v = std::move(big)]() { return v.size(); };  // 移动
auto f = [&big]() { return big.size(); };  // 引用(注意生命周期)
```

## 仿函数仍有优势的场景

| 场景 | 理由 |
|------|------|
| 跨编译单元共享 | lambda 是局部的,仿函数可以放在头文件共享 |
| 需要命名 | 调试时能看到类名,lambda 内部名是乱码 |
| 需要继承/多态 | 仿函数可以继承自基类,实现多态行为 |
| 显式控制成员函数 | 仿函数可以显式控制拷贝/移动,lambda 不行 |
| 类型作为模板参数 | 仿函数是 named type,可以用作 NTTP |

## lambda 能完全替代仿函数吗?

**日常开发中 99% 的场景可以**。lambda 更简洁、就地定义、不需要额外命名。但仿函数在**需要跨编译单元复用、需要继承体系、需要精细控制对象语义**时仍有价值。

&gt; **选择标准是可读性和维护性,不是性能**。

## 函数指针 vs lambda

| 维度 | 函数指针 | 无捕获 lambda | 有捕获 lambda |
|------|---------|--------------|--------------|
| 转换为函数指针 | 是 | **是** | ❌ |
| 携带状态 | ❌ | ❌ | ✅ |
| 性能 | 优(直接调用) | 优 | 优(可内联) |

**无捕获的 lambda 可以隐式转换为函数指针**(因为它没有状态,等价于普通函数)。但**有捕获的 lambda 不能**——它有状态(成员变量),需要一个对象来承载。

这就是为什么 C 风格回调接口(如 `qsort`)通常需要一个额外的 `void* userdata` 参数来传递上下文。

## 面试高频追问

**Q1: `[=]` 和 `[&]` 哪个更好?**
**没有"更好"**。两者都有陷阱:`[=]` 可能意外拷贝大对象(性能问题),`[&]` 可能悬垂引用(正确性问题)。**最佳实践**:显式列出每个捕获(`[x, &y]`),让代码意图明确。

**Q2: lambda 的开销在哪里?**
**唯一有开销的场景**:
- 通过 `std::function` 传递(类型擦除)
- 用作 `std::function` 成员
- 作为函数参数按值传递大对象捕获(拷贝)

否则 lambda 和普通函数一样快。

**Q3: lambda 递归怎么写?**
需要 `std::function` 包装自己:

```cpp
std::function<int(int)> fib = [&fib](int n) {
    return n < 2 ? 1 : fib(n-1) + fib(n-2);
};
```

或者用 Y-combinator 模式(避免 std::function 开销)。

**Q4: lambda 和仿函数哪个更"现代"?**
**lambda 是趋势**。C++20 引入 `template <typename... Ts> concept` 让 lambda 可以作为模板参数(以前需要手写仿函数)。**但仿函数在性能敏感场景仍有不可替代的价值**(确定性大小、零开销)。
