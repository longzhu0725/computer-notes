# 智能指针实现原理

## 核心结论

智能指针本质是一个**类模板**,封装原始指针并重载 `*` 和 `->` 操作符,利用 [RAII 机制](/notes/raii-机制.html)在对象析构时自动释放资源。三种智能指针分别通过**独占所有权、共享所有权、弱引用**来管理不同的生命周期场景。

| 智能指针 | 所有权模型 | 实现机制 | 大小 | 性能开销 |
|---------|-----------|---------|------|---------|
| `unique_ptr` | **独占** | 禁拷贝 + 移动 | 单指针大小(默认) | 0(零开销抽象) |
| `shared_ptr` | **共享** | 引用计数 + 控制块 | 2 倍指针大小 | 引用计数原子操作 |
| `weak_ptr` | **弱引用** | 共享控制块 + 不增加强引用计数 | 2 倍指针大小 | lock() 原子检查 |

## unique_ptr:独占所有权

实现上很简单:删除拷贝构造和拷贝赋值,只保留移动语义。同一时刻只有一个 unique_ptr 拥有资源,析构时直接 `delete`。**零开销,性能和裸指针一样**。

```cpp
template <typename T>
class unique_ptr {
    T* ptr_;
public:
    explicit unique_ptr(T* p = nullptr) : ptr_(p) {}
    ~unique_ptr() { delete ptr_; }
    // 禁拷贝
    unique_ptr(const unique_ptr&) = delete;
    unique_ptr& operator=(const unique_ptr&) = delete;
    // 允移动
    unique_ptr(unique_ptr&& other) noexcept : ptr_(other.ptr_) { other.ptr_ = nullptr; }
    unique_ptr& operator=(unique_ptr&& other) noexcept {
        if (this != &other) { delete ptr_; ptr_ = other.ptr_; other.ptr_ = nullptr; }
        return *this;
    }
    // 访问
    T& operator*() const { return *ptr_; }
    T* operator->() const { return ptr_; }
    T* get() const { return ptr_; }
    T* release() { T* p = ptr_; ptr_ = nullptr; return p; }
};
```

**大小**:默认是 1 个指针(8 字节)。如果指定 `unique_ptr<T[]>` 是另一种特化(支持 `operator[]`)。

## shared_ptr:共享所有权

核心是一个**控制块(Control Block)**,里面存两个原子计数器:

- `shared_count`:**强引用数**(决定对象生命周期)
- `weak_count`:**弱引用数**(决定控制块生命周期)

```cpp
struct ControlBlock {
    std::atomic<long> shared_count{1};
    std::atomic<long> weak_count{1};  // 1 表示控制块自身
    // 删除器(deleter)如果有
};
```

```
                ┌────────────────────┐
shared_ptr ─────►  ptr_  (data)  ◄─── weak_ptr
                └────────────────────┘
                              │
                              ▼
                ┌────────────────────┐
                │   ControlBlock     │
                │  - shared_count    │
                │  - weak_count      │
                │  - deleter         │
                │  - allocator       │
                └────────────────────┘
```

**生命周期规则**:

1. 拷贝 `shared_ptr` → `shared_count` 原子 +1
2. 析构 `shared_ptr` → `shared_count` 原子 -1
3. `shared_count` 归 0 → 释放**资源对象**(调用 deleter)
4. `weak_count` 归 0 → 释放**控制块**

## weak_ptr:弱引用

和 shared_ptr 共享同一个控制块,但**不增加 shared_count**。通过 `lock()` 尝试提升为 shared_ptr:

```cpp
std::shared_ptr<T> sp = wp.lock();  // 尝试提升
if (sp) {
    // 对象还活着,sp 可以安全使用
} else {
    // 对象已销毁,sp 为空
}
```

**如果 shared_count 已经为 0,lock() 返回空,说明对象已销毁**。这就是它能解决循环引用的原因:两个对象互相持有 weak_ptr 不会阻止对方析构。

```cpp
// 循环引用问题:shared_ptr 互持,内存泄漏
struct Node {
    std::shared_ptr<Node> next;  // 改成 weak_ptr 即可解决
    std::weak_ptr<Node> prev;    // weak_ptr 不增加强引用计数
};
```

## make_shared 的优势

`make_shared<T>(args)` 一次分配就搞定**对象和控制块**(放在同一块内存里),而 `new T` + `shared_ptr` 构造需要两次分配。

```cpp
// 两次分配:new T + 控制块
std::shared_ptr<T> sp1(new T(args...));

// 一次分配:对象和控制块连续
auto sp2 = std::make_shared<T>(args...);
```

| 维度 | new + shared_ptr | make_shared |
|------|-----------------|-------------|
| 分配次数 | 2 次 | 1 次 |
| 缓存友好 | 差(2 个 block) | 好(连续) |
| 异常安全 | 中(可能泄漏 new 出来的对象) | 优(原子) |
| 对象内存释放时机 | 控制块释放时 | 最后一个 shared_ptr 析构时 |
| 控制块内存释放时机 | 最后一个 weak_ptr 析构时 | 最后一个 weak_ptr 析构时 |

**注意**:make_shared 的小代价——对象内存和控制块**绑在一起**,weak_ptr 不死,对象那块内存也回收不了(因为 weak_ptr 的 lock() 需要访问控制块,而控制块和对象相邻,不能单独释放)。

## 控制块什么时候释放

资源对象在 `shared_count` 归零时释放,但**控制块要等 `weak_count` 也归零才释放**。所以:

- 如果有 weak_ptr 还活着,控制块这块内存会一直留着
- 对象本身(被 shared_count 归零时释放)可以正常释放
- 但 `make_shared` 的话,对象内存和控制块**绑在一起**,所以对象也回收不了

## unique_ptr 与 shared_ptr 的转换

| 转换方向 | 是否合法 | 备注 |
|---------|---------|------|
| `unique_ptr` → `shared_ptr` | ✅ | `shared_ptr<T> sp = std::move(up);` 单向的所有权放大 |
| `shared_ptr` → `unique_ptr` | ❌ | 不可能缩小,会泄漏 |

**为什么 unique_ptr → shared_ptr 可以?** 因为 unique_ptr 持有独占资源,转移给共享池是"放大"语义,系统可以构造控制块接管。

**为什么 shared_ptr → unique_ptr 不行?** shared_ptr 的引用计数可能 &gt; 1,转移给独占就破坏了共享语义,会导致另一个 shared_ptr 析构时 double-free。

## 智能指针的线程安全

分三个层面:

1. **控制块(引用计数)**:线程安全(`std::atomic`)
2. **shared_ptr 实例本身**:不安全(同一变量多线程写需加锁)
3. **管理的对象**:不安全(智能指针只管生命周期,不管对象内部并发)

```cpp
// 多线程下安全的典型用法
void worker(std::shared_ptr<Data> sp) {  // 按值传递,各线程独立副本
    sp->mutate();  // 仍需 mutex 保护 Data 内部
}
```

## 面试高频追问

**Q1: 为什么 unique_ptr 零开销?**
- 没有引用计数(独占)
- 没有控制块(在编译期被消化)
- `*` 和 `->` 操作符可以被内联
- 析构调用的是普通 delete,不进 vtable

**Q2: shared_ptr 的循环引用怎么破?**
- 用 weak_ptr 替代其中一个方向
- 引入第三方节点作为中介
- 业务层面避免互相拥有

**Q3: 智能指针能管理数组吗?**
- `unique_ptr<T[]>`:支持,析构时调用 `delete[]`
- `shared_ptr<T[]>`:C++17 起支持
- 但更推荐用 `std::vector<T>` 或 `std::array<T, N>`,表达更清晰

**Q4: enable_shared_from_this 是什么?**
在类的成员函数中安全地获取指向自己的 `shared_ptr` 的机制:

```cpp
class Widget : public std::enable_shared_from_this<Widget> {
public:
    std::shared_ptr<Widget> getptr() {
        return shared_from_this();  // 安全获取 this 的 shared_ptr
    }
};
```

**前提**:对象必须已经由 shared_ptr 管理(否则 `shared_from_this()` 抛异常)。
