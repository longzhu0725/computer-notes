# 构造函数与析构函数的虚函数规则

## 核心结论

- **构造函数不能是虚函数**——构造期间 vptr(虚表指针)尚未建立,虚函数机制无法工作
- **析构函数应该是虚函数**(基类场景)——确保 `delete base_ptr` 时调用整个析构链,避免派生类资源泄漏

| 函数 | 能否为虚 | 原因 | 调用顺序 |
|------|---------|------|---------|
| 构造函数 | **不能** | 构造期间 vptr 未建立,虚函数机制不起作用 | 基类 → 成员 → 派生类 |
| 析构函数 | **可以,基类应声明** | 不声明则 `delete base_ptr` 只调基类析构,派生类资源泄漏 | 派生类 → 成员 → 基类 |

## 一、vptr 的生命周期

### 构造过程中的 vptr 变化

```
1. 分配对象内存(sizeof(Derived) 大小)
2. vptr 指向 Base 的虚表 ← 此时对象是"Base"
3. 执行 Base 构造函数体
4. vptr 指向 Derived 的虚表 ← 此时对象是"Derived"
5. 执行 Derived 构造函数体
```

构造分阶段:每一层基类构造时,对象"暂时"是当前正在构造的类。vptr 跟随构造进度切换,确保构造期间调用任何虚函数都静态绑定到当前类版本。

### 析构过程中的 vptr 变化

```
1. 进入 Derived 析构函数体,vptr → Derived 虚表
2. Derived 析构体执行完
3. 析构 Derived 成员变量
4. vptr 切换回 Base 虚表 ← 此时对象是"Base"
5. 执行 Base 析构函数体
6. 释放对象内存
```

析构时 vptr 也跟随析构进度切换——保证析构期间虚函数调用不会"穿透"到已销毁的派生类部分。

## 二、构造函数不能为虚函数的根本原因

### 虚函数机制依赖 vtable

```cpp
Base* p = new Derived();  // 编译期:Base;运行期:Derived
p->virtualFunc();          // 查 vtable,调用 Derived::virtualFunc
```

这个机制要求 `p` 指向的对象**已构造完成**——vptr 已就绪,虚表已指向正确的实现。

### 构造期间 vptr 状态

构造的第一件事是**建立 vptr**。在 vptr 建立之前:
- 对象还不"完整"——只有基类子对象
- 虚表未就绪——编译器无法确定"该调用哪个版本"
- 强行调用虚函数→只能静态绑定到当前类(无法多态)

```cpp
class Base {
public:
    Base() {
        virtualFunc();  // 编译期绑定到 Base::virtualFunc,不是 Derived
    }
    virtual void virtualFunc() { std::cout << "Base"; }
};

class Derived : public Base {
    int* data_;
public:
    Derived() : data_(new int[100]) {}  // 派生类成员还未构造
    void virtualFunc() override { std::cout << "Derived"; }
    ~Derived() { delete[] data_; }
};

Derived d;  // 输出 "Base",不是 "Derived"!
```

**为什么?**:构造 `Derived` 时,先构造 `Base`,vptr 指向 `Base` 虚表,然后才构造 `Derived` 成员和切换 vptr。`Base()` 体内的 `virtualFunc()` 调用发生在切换之前,只能找到 `Base` 的版本。

## 三、为什么基类析构函数必须是虚函数

### 不虚析构的内存泄漏

```cpp
class Base {
public:
    ~Base() { std::cout << "~Base\n"; }  // 警告:有派生类应声明为 virtual
};
class Derived : public Base {
    int* data_;
public:
    Derived() : data_(new int[100]) {}
    ~Derived() { delete[] data_; std::cout << "~Derived\n"; }
};

Base* p = new Derived();
delete p;
// 输出:~Base
// 问题:Derived 的 data_ 没释放!内存泄漏
```

### 虚析构的修复

```cpp
class Base {
public:
    virtual ~Base() { std::cout << "~Base\n"; }  // 关键:virtual
};
// 同样代码 delete p;
// 输出:~Derived → ~Base
// 正确:派生类析构先调,基类析构后调
```

### 编译器警告

现代编译器会主动警告:`'Base' has virtual functions, but the destructor is not virtual`。看到此警告:**加 `virtual`**。

## 四、何时不需要虚析构

### 不作为基类的类不需要

```cpp
class Point {       // 不会被继承
    int x_, y_;
public:
    ~Point() = default;  // 不需要 virtual,徒增 vtable 开销
};
```

### 不通过基类指针删除时不需要

```cpp
Derived d;          // 栈对象
// d 的析构一定调 Derived::~Derived,不需要 virtual

Derived* p = new Derived();
delete p;            // 编译期类型就是 Derived*,不需要 virtual
```

**判断标准**:类是否**预期被继承**,且派生类对象**可能通过基类指针/引用管理**(存储在容器中、作为接口返回等)。满足两个条件→虚析构。

## 五、纯虚析构函数(Pure Virtual Destructor)

### 让类成为抽象类

```cpp
class AbstractBase {
public:
    virtual ~AbstractBase() = 0;  // 纯虚析构
};
// 错误!纯虚函数必须提供实现
// AbstractBase ab;  // 不能实例化(抽象类)
```

### 纯虚析构也需要实现

```cpp
class AbstractBase {
public:
    virtual ~AbstractBase() = 0;
};
AbstractBase::~AbstractBase() = default;  // 必须在类外提供实现

class Derived : public AbstractBase {};
Derived d;  // OK
// 析构时:Derived::~Derived → AbstractBase::~AbstractBase
```

**为什么需要实现**:基类析构总是会被调用(派生类析构最终会回到基类),没有实现会导致链接错误。

## 六、构造函数中调用虚函数

```cpp
class Base {
public:
    Base() { init(); }       // 实际是 Base::init,非多态
    virtual void init() { std::cout << "Base::init\n"; }
};
class Derived : public Base {
public:
    Derived() { }            // 此时 init() 已被绑定
    void init() override { std::cout << "Derived::init\n"; }
};

Derived d;  // 输出 "Base::init",非 "Derived::init"
```

**正确做法**:不要在构造函数中调用虚函数。如果需要"派生类特定的初始化",用**两阶段初始化**(构造函数只做成员初始化,初始化逻辑放在 `init()` 虚函数中,显式调用):

```cpp
class Base {
    bool inited_ = false;
public:
    Base() = default;
    virtual void init() {  // 派生类可重写
        // 通用初始化
        inited_ = true;
    }
};

class Derived : public Base {
public:
    void init() override {
        Base::init();  // 先调基类
        // 再做派生类特定初始化
    }
};

// 使用
Derived d;
d.init();  // 显式调用,此时多态机制完全就绪
```

## 七、析构函数中抛出异常

### 析构异常的双重灾难

```cpp
class Bad {
public:
    ~Bad() { throw std::runtime_error("destructor throws"); }
};

void f() {
    Bad b;          // 构造
    throw 1;        // 触发栈展开
}                   // 析构 b 时又抛异常 → std::terminate!
```

栈展开时,如果析构函数也抛异常,程序必须调用 `std::terminate` 直接终止(C++ 标准规定无法处理两个同时活跃的异常)。

### 正确做法:析构吞掉异常

```cpp
class File {
    FILE* fp_;
public:
    ~File() noexcept {  // 显式 noexcept
        if (fp_) {
            if (fclose(fp_) != 0) {
                // 错误处理:不抛异常,记日志或吞掉
                log_error("fclose failed");
            }
        }
    }
};
```

C++11 起,所有析构函数**默认 `noexcept`**,即使不显式标注。如果你写 `~T() noexcept(false)`,编译器会发出警告。

## 八、面试高频追问

**Q1: 什么时候必须用虚析构?**
当类**预期被继承**,且派生类对象**可能通过基类指针/引用删除**。两个条件同时满足,析构就必须 `virtual`。典型场景:工厂函数返回基类指针、容器存基类指针(虽然现在推荐 `unique_ptr<Base>`)、接口类的析构。

**Q2: 构造函数中调用虚函数会怎样?**
编译期绑定,非多态。`Base()` 体内调 `virtualFunc()`,只会调 `Base::virtualFunc`,因为构造期间派生类部分还未构造,vptr 指向的是当前类的虚表。**最佳实践**:构造函数内不要调虚函数,用两阶段初始化或 `final` 标识类禁止继承。

**Q3: 析构函数中抛异常会怎样?**
双重异常导致 `std::terminate`。栈展开时如果析构抛异常,标准库无法处理两个同时活跃的异常,直接终止程序。**最佳实践**:析构函数 `noexcept`(默认),内部 `try-catch` 吞掉所有异常,或留待应用层显式调用 `close()`。

**Q4: `final` 关键字和虚析构的关系?**
`class Derived final : public Base {}` 表示 `Derived` 不可被继承。此时基类析构**可以不是 virtual**——因为不可能有"通过基类指针删除未知的派生类"的场景。`final` 是一种优化手段,可以避免不必要的 vtable 开销。

**Q5: 抽象基类的析构函数怎么写?**
写成**纯虚析构**可以强制类为抽象:`virtual ~Base() = 0;`,但**必须在类外提供实现**:`Base::~Base() = default;`。这是少有的"纯虚函数必须实现"的情况,因为基类析构总是会被调用。
