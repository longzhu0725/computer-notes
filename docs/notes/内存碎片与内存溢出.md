# 内存碎片与内存溢出

## 核心结论

**内存碎片**是系统中空闲内存总量够用,但因为不连续(外部碎片)或分配粒度浪费(内部碎片)而无法满足分配请求的现象。**栈溢出**是栈空间被耗尽的错误(无限递归把栈帧堆满了);**堆溢出**是往动态分配的内存块之外写数据的错误(本质是越界访问,常被利用做攻击)。两者一个是"**装不下了**",一个是"**写出界了**"。

| 维度 | 栈溢出(Stack Overflow) | 堆溢出(Heap Overflow) |
|------|----------------------|---------------------|
| 发生区域 | 栈内存(向低地址增长) | 堆内存(动态分配区) |
| 根本原因 | 空间耗尽:递归过深或局部变量过大 | 越界写入:往 `malloc`/`new` 块之外写 |
| 触发方式 | 编程错误(无限递归、栈上大数组) | 可无意错误,也常是恶意攻击 |
| 后果 | 程序立即崩溃(SIGSEGV),确定性 | 可能立即崩溃,也可能潜伏——取决于覆盖了什么 |
| 安全影响 | 可用性(程序挂了) | 安全性(可能执行任意代码) |

## 内存碎片

### 外部碎片(External Fragmentation)

频繁分配释放不同大小的内存块后,空闲内存被切成很多**不连续**的小块。单个小块都不够大,但加起来总量足够——"**有钱但花不出去**"。

C/C++ 中特别严重,因为**指针直接指向内存地址**,没法像 Java 那样做内存压缩(移动对象需要更新所有指针,几乎不可能)。

### 内部碎片(Internal Fragmentation)

分配器按**固定粒度**分配(比如 8 字节对齐),你要 5 字节它给你 8 字节,多出来的 3 字节就浪费了。**内存池**也有这个问题——池里的块大小固定,小请求浪费空间。

| 类型 | 成因 | 解决方案 | C++ 难度 |
|------|------|---------|---------|
| 外部碎片 | 分配释放顺序不连续 | buddy system、best-fit、slab | **很难**(指针语义) |
| 内部碎片 | 分配粒度&gt;实际需求 | 选合适的粒度、内存池块大小 | 容易(选型) |

## 外部碎片为什么比内部碎片难解决?

**内部碎片**是"已知的浪费",上限可控——选合适的分配粒度或内存池块大小就行。

**外部碎片**是"**全局的混乱**",取决于分配释放的随机序列,没法预测。解决它要做**内存压缩**(移动已分配块来合并空闲块),但 C/C++ 里指针是裸地址,移动了对象就得更新所有指向它的指针——**几乎不可能**。所以只能靠好的分配策略(best-fit、buddy system)来缓解。

## 栈溢出 vs 堆溢出

### 栈溢出

栈是一段**连续的内存区域**,默认大小 1-8MB(Linux 默认 8MB,Windows 默认 1MB)。无限递归或栈上分配大数组会撑爆栈空间:

```cpp
void infinite_recursion() {
    char buf[1024];          // 每帧 1KB
    infinite_recursion();     // 无限递归
}
```

栈向下增长,栈溢出通常会**立即**触发 SIGSEGV 崩溃。

### 堆溢出

堆是**动态分配**的内存,通过指针访问。堆溢出就是越界写入堆上的缓冲区:

```cpp
char* buf = (char*)malloc(10);
strcpy(buf, "0123456789");   // 写 11 字节,溢出 1 字节
```

堆溢出的危害**远大于**栈溢出:

- 覆盖相邻对象的 vtable → **控制流劫持**
- 覆盖堆元数据 → 任意地址读写
- 配合 ROP/JOP 链 → **任意代码执行**

这就是为什么堆溢出是**严重的安全问题**,而栈溢出只是"程序 bug"。

## 堆溢出为什么是严重的安全问题

堆溢出让攻击者能**覆盖堆里相邻的数据**。精心构造溢出内容可以:

- 覆盖**函数指针**让它指向恶意代码
- 覆盖对象的 **vtable** 控制执行流
- 修改相邻数据改变程序逻辑
- 配合 **ROP 链**,即使开了 DEP/NX 保护也能执行任意代码

**真实案例**:Heartbleed(心脏滴血)漏洞就是 OpenSSL 的堆缓冲区过度读取,导致私钥泄漏。

## 如何避免栈溢出

| 方法 | 说明 |
|------|------|
| 递归改迭代 | 用 `std::stack` 或循环显式控制深度 |
| 不在栈上放大数组 | 改用堆分配(`std::vector`) |
| 给递归设深度上限 | 显式 depth counter |
| 编译时开 `-fstack-protector` | 检测栈溢出覆盖 |
| 调整栈大小 | `ulimit -s 65536`(临时) |

```cpp
// 错误:递归遍历百万节点链表
void process(Node* n) {
    if (n) { process(n->next); }  // 栈深度 ≈ 节点数,溢出
}

// 正确:循环
void process(Node* head) {
    while (head) { head = head->next; }
}
```

## 所有递归都会栈溢出吗?

**不会**。只要深度在栈容量内就没事。

- 遍历深度 1000 的二叉树:栈深度 ≈ 树高(平衡树 log₂1000 ≈ 10),**没问题**
- 递归遍历百万节点链表:栈深度 ≈ 节点数(百万),**会溢出**
- 尾递归:可以被编译器优化成循环(`-O2` 下 GCC/Clang),不增长栈空间

## 调试工具

| 工具 | 检测栈溢出 | 检测堆溢出 | 检测碎片 | 性能损耗 |
|------|----------|----------|---------|---------|
| Valgrind | ✅ | ✅ | ✅ | 10-50x |
| AddressSanitizer | ✅ | ✅ | 部分 | 2x |
| `-fstack-protector` | ✅ | ❌ | ❌ | &lt;1% |
| `mtrace` | ❌ | ❌ | 部分 | 低 |
| `glibc mallinfo` | ❌ | ❌ | ✅ | 无 |

**推荐组合**:开发期 AddressSanitizer + UBSan,生产期 LeakSanitizer(仅泄漏)。

## 面试高频追问

**Q1: 内部碎片和外部碎片哪个对性能影响大?**
**外部碎片**。内部碎片的总量是"浪费率 × 总分配量",可控且有上限(选粒度就行)。外部碎片可能让总空闲够用但分配失败,**严重影响程序可用性**。

**Q2: 内存池一定能解决碎片吗?**
**不能**。内存池按**固定大小**分配,只能解决"小对象频繁分配释放"的碎片,对**变长对象**反而可能加剧碎片(大对象用不完池里的块)。

**Q3: GC 能解决外部碎片吗?**
部分解决。GC 可以移动对象并更新引用(JVM 的标记-整理算法),**这是 C/C++ 做不到的**——所以 C++ 必须靠分配器策略。

**Q4: 栈溢出能像堆溢出那样被攻击吗?**
**能**,但难度不同。栈溢出可覆盖返回地址劫持控制流(经典栈溢出攻击)。**但现代编译器默认开启**:
- 栈金丝雀(`-fstack-protector`)
- ASLR(地址随机化)
- DEP/NX(栈不可执行)

使攻击难度大幅提高。
