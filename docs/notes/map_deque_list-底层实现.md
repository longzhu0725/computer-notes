# map/deque/list 底层实现

## 核心结论

| 容器 | 底层结构 | 随机访问 | 两端插入 | 中间插入 | 迭代器失效 | 缓存性能 |
|------|---------|---------|---------|---------|----------|---------|
| `map` | **红黑树**(自平衡 BST) | 不支持 | O(log n) | O(log n) | 删除当前才失效 | 差 |
| `deque` | **分段连续数组**(中控 map + buffer) | **O(1)** | **O(1)** | O(n) | 头尾插入不失效,中间失效 | 较好 |
| `list` | **双向链表** | 不支持 | O(1) | O(1)(已知位置) | 删除当前才失效 | 差 |

## map:红黑树

红黑树是自平衡二叉搜索树,保证最长路径不超过最短路径的两倍,所有操作 O(log n)。**键自动有序**(按 `<` 运算符或自定义比较器)。

**关键特性**:

- 插入/删除**不会导致其他迭代器失效**——树结构的天然优势(节点独立)
- 范围查询、按序遍历用 map
- `unordered_map` 用哈希表,平均 O(1) 但最坏 O(n),rehash 时所有迭代器失效

## deque:分段连续存储

核心结构:**一个指针数组**(中控 map),每个指针指向一个**固定大小的连续缓冲区**。

```
+-----------------------+
|   中控 map (指针数组)  |
|   [0] [1] [2] [3]     |
|    |   |   |   |       |
|    v   v   v   v       |
|   [_ _ _][_ _ _][_ _ _]|
|   [_ _ _][_ _ _]      |  <- 每个 buffer 固定大小(如 512 元素)
+-----------------------+
```

**插入策略**:`push_front`/`push_back` 只需在当前块的前/后填充,块满了就分配新块并在中控 map 中加一个指针。**不需要搬迁已有元素**。

**随机访问 O(1) 的奥秘**:
1. 算目标在第几个块:`offset / block_size`
2. 算块内位置:`offset % block_size`
3. 通过中控 map 取 buffer 指针 + 块内偏移

**两次除法**就完成定位——所以是 O(1) 随机访问。

**迭代器失效规则**:

- 头尾插入:不失效
- 中间插入:首尾迭代器可能失效(可能跨块移动)
- 删除:类似 vector,被删元素之后的迭代器失效

## list:双向链表

每个节点独立分配,通过前驱/后继指针串联:

```cpp
template <typename T>
struct _List_node {
    _List_node* _prev;
    _List_node* _next;
    T _data;
};
```

**关键特性**:

- 已知迭代器位置时,插入/删除只需改指针,O(1)
- 定位第 n 个元素必须从头遍历,O(n)
- 节点分散在堆上,**缓存不友好**(cache miss 多)
- 插入/删除**不会导致其他迭代器失效**——这是 list 在某些场景下不可替代的原因

**list 在现代硬件上性能差的主要原因**是**缓存局部性差**——节点分散,每次访问都是 cache miss。遍历 100 万个 list 节点比 vector 慢 10 倍以上。

## 选型决策

| 场景 | 推荐容器 | 理由 |
|------|---------|------|
| 频繁随机访问 | `vector` | 缓存最优 |
| 头尾插入/删除 + 偶尔随机访问 | `deque` | 两端 O(1),中间随机访问 O(1) |
| 频繁在中间插入/删除,需迭代器稳定 | `list` | 中间插入 O(1) 不失效 |
| 需要键有序 + 范围查询 | `map` | 红黑树天然有序 |
| 只需快速查找,不关心顺序 | `unordered_map` | 哈希 O(1) |
| 频繁头插 | `deque` 或 `list` | vector 头插 O(n) |
| 大量小对象 + 频繁分配释放 | `deque` / `vector` | 缓存友好 |

**经验法则**:

- 默认用 `vector`(缓存最优)
- 需要稳定迭代器 + 中间插入 → `list`
- 需要两端操作 + 随机访问 → `deque`
- 需要键有序 → `map`;只关心查找 → `unordered_map`

## deque 中间插入为什么比 list 慢?

deque 中间插入需要**移动元素来维持块内连续性**:

- 平均移动 n/2 个元素
- 还可能跨块移动
- 总成本 O(n)

list 只需改**两个指针**,O(1) 恒定时间。

但实际场景中,如果插入不频繁,deque 的**缓存友好性**带来的遍历优势远超 list:

- 遍历 100 万个 int:vector &lt; deque &lt; list(差距 1 个数量级)
- 100 万次中间插入:list &lt; deque(差距 1-2 个数量级)

**结论**:中间插入 &lt; 10% 时 deque 综合更优;&gt; 50% 时 list 才有明显优势。

## 什么时候该用 list 而不是 deque?

**两种情况**:

1. 需要**频繁在中间位置插入/删除**(且已持有迭代器)
2. 需要**迭代器/指针/引用在插入删除后不失效**

除此之外,现代 C++ 中 `deque` 甚至 `vector` 几乎总是比 `list` 快,因为**缓存局部性**的优势太大了。

## deque 能完全替代 vector 吗?

**不能**。deque 的内存不是完全连续的(分段连续),**不能直接传给需要连续内存的 C API**。vector 保证内存连续,可以用 `data()` 拿到裸指针。

| 与 C API 互操作 | vector | deque |
|---------------|--------|-------|
| `data()` 拿到连续指针 | ✅ | ❌(分段) |
| 传给 C 函数(如 `qsort`) | ✅ | ❌ |
| 性能 | 优 | 较好(块内连续,跨块跳转) |

另外 vector 的缓存性能**比 deque 更好**(完全连续 vs 分段连续)。deque 是"想要两端插入又想要随机访问但又不强制连续内存"的折中。

## 面试高频追问

**Q1: 红黑树和 AVL 树哪个更适合做 map 底层?**
**红黑树**。虽然 AVL 树更严格平衡(高度差 ≤ 1),查找更快,但每次插入/删除都可能需要旋转调整。红黑树**近似平衡**(高度差 ≤ 2 倍),调整次数更少,**插入/删除性能更优**。map 更看重插入删除,选红黑树。

**Q2: list 的 size() 是 O(1) 吗?**
C++11 之前是 O(n)(需要遍历计数),C++11 之后是 O(1)(维护了 size 字段)。这是 C++11 的**重大修正**,之前代码中 `list.size() == 0` 性能很差。

**Q3: deque 随机访问是 O(1),为什么不用 deque 替代 vector?**
- deque 的**两次除法**比 vector 的一次乘法慢
- 跨块访问的 cache miss 比 vector 多
- 与 C API 不兼容
- 大多数场景只有尾插,vector 更合适

**Q4: 节点式容器(list/map)的 erase 复杂度?**
- list:已知位置 O(1),但 erase(it) 返回下一个迭代器(因当前节点失效)
- map/unordered_map:erase(it) 均摊 O(1)(O(log n) 找 + O(1) 删除)
