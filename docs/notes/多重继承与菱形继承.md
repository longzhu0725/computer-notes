# 多重继承与菱形继承

## 核心结论

多重继承让一个类同时继承多个基类,好处是组合多个接口,坏处是当多个基类有同名成员时产生**二义性**;更严重的是**菱形继承**问题:`D` 继承 `B` 和 `C`,而 `B` 和 `C` 都继承自 `A`,导致 `D` 里有**两份** `A` 子对象。C++ 用**虚继承**解决:`B` 和 `C` 虚继承 `A`,编译器保证 `D` 里只有一份 `A`,通过虚基类指针(`vbptr`)间接访问。

| 维度 | 普通多重继承 | 虚继承 |
|------|------------|--------|
| 菱形继承时 `A` 子对象数量 | 2 份 | 1 份 |
| 访问 `A` 成员 | 需要 `d.B::name` 消歧义 | 直接 `d.name` |
| 内存布局 | 嵌入两份 `A` | 一份 `A` + 多个 `vbptr` |
| 访问性能 | 直接寻址 | 间接寻址(查 vbtable) |
| 构造责任 | 各路径单独构造 | 最派生类单独构造 |

## 菱形继承到底出了什么问题

普通多重继承下,`D` 的对象内存里包含一个完整的 `B` 子对象和一个完整的 `C` 子对象。而 `B` 里有一份 `A`,`C` 里也有一份 `A`——所以 `D` 里有两份 `A`。这带来两个问题:

1. **浪费内存**:`A` 的数据存了两份
2. **二义性**:访问 `A` 的成员时编译器不知道该用 `B` 路径的那份还是 `C` 路径的那份

```cpp
class A { public: int x; };
class B : public A {};
class C : public A {};
class D : public B, public C {};  // D 里有两份 A

D d;
d.x = 10;       // 错误:二义性,'x' 在 B::A 和 C::A 中都有
d.B::x = 10;    // 显式消歧义
```

## 虚继承怎么解决

`B` 和 `C` 用 `virtual public A` 继承时:

1. 编译器不再把 `A` 的子对象直接嵌入 `B` 和 `C` 内部
2. 而是在 `B` 和 `C` 里各放一个**虚基类指针**(`vbptr`)
3. `vbptr` 指向**虚基类表**(`vbtable`),表里记录 `A` 子对象相对于当前对象的偏移量
4. 最终 `D` 的内存布局里只有一份 `A`,`B` 和 `C` 通过各自的 `vbptr` 间接访问这份共享的 `A`

**代价**:访问虚基类成员多了一次间接寻址,对象体积也多了 `vbptr` 的开销(每个虚继承路径 4/8 字节)。

## 为什么虚基类要由最派生类初始化

普通继承下,`B` 的构造函数负责初始化它的 `A` 子对象。但虚继承下 `A` 只有一份,如果 `B` 和 `C` 都去初始化就冲突了。C++ 规定:

> **虚基类由最派生类的构造函数直接初始化,中间基类构造函数里对 `A` 的初始化会被忽略。**

```cpp
class A { public: A(int v) : x(v) {} int x; };
class B : virtual public A { public: B() : A(1) {} };  // A(1) 被忽略
class C : virtual public A { public: C() : A(2) {} };  // A(2) 被忽略
class D : public B, public C {
public:
    D() : A(3), B(), C() {}  // 实际调用 A(3),B 和 C 的 A() 都被忽略
};
```

## 多重继承 vs 组合

| 维度 | 多重继承 | 组合 |
|------|---------|------|
| 关系语义 | "is-a"(派生类是基类) | "has-a"(派生类拥有基类对象) |
| 适用场景 | 接口组合(纯虚基类) | 复用基类实现 |
| 内存布局 | 子对象嵌入 + 可能的 vtable | 独立成员对象 |
| 菱形问题 | 需要虚继承解决 | 完全避免 |
| 耦合度 | 高(派生类依赖基类细节) | 低(只暴露接口) |

**经验法则**:

- 继承表达"is-a"关系
- 组合表达"has-a"关系
- **基类有数据成员时优先组合**;只有纯虚接口才用多重继承

## 名称冲突怎么解决

用作用域解析运算符显式指定,或在派生类里重写同名函数:

```cpp
class B { public: void f() { std::cout << "B\n"; } };
class C { public: void f() { std::cout << "C\n"; } };
class D : public B, public C {
public:
    void f() { B::f(); C::f(); }  // 二者都调用
};
```

## 多重继承时析构函数的注意点

- 基类析构函数**必须**声明为 `virtual`
- 否则通过基类指针 `delete` 派生类对象时,不会调用派生类析构函数,导致资源泄漏
- 编译器会按继承声明的逆序自动调用各基类析构函数,不需要手动调用

## Java/Go 为什么不支持多重继承

| 语言 | 替代方案 | 设计动机 |
|------|---------|---------|
| Java | 一个类 + 多个 interface | 避免菱形继承 |
| Go | 组合 + interface,完全无继承 | 接口隐式实现,无类型层级 |
| C++ | 多重继承 + 虚继承 | 灵活但易滥用 |

C++ 保留多重继承是为了灵活性,但实际工程中**建议只在接口组合场景使用**——即所有基类都是纯虚接口(没有数据成员),用接口表达"能做某事"的能力。

## 面试高频追问

**Q1: 虚继承的对象布局具体什么样?**
以 `D : public B, public C`(`B`、`C` 都虚继承 `A`)为例,典型布局:

```
+-----------------+
| vptr (B 部分)   | -> B 的 vtable
| B 数据成员      |
| vbptr_B         | -> 虚基类表(指向 A 偏移)
| vptr (C 部分)   | -> C 的 vtable
| C 数据成员      |
| vbptr_C         | -> 虚基类表(指向 A 偏移)
+-----------------+
| A 数据成员      | <- 共享的 A 子对象
+-----------------+
```

**Q2: 虚基类表存什么?**
存的是**虚基类子对象相对于当前对象起始地址的偏移量**。访问 `d.B::A::x` 时,通过 `d` 找到 `vbptr_B`,查表得 `A` 偏移 `offset`,然后 `*(d + offset + x_offset)`。

**Q3: 为什么 `iostream` 同时继承 `istream` 和 `ostream`?**
`iostream` 虚继承 `ios_base`,`istream` 和 `ostream` 都虚继承 `ios_base`。这样 `iostream` 里只有一份 `ios_base`(streambuf 等共享成员)。这是 C++ 标准库里虚继承的典型案例。
