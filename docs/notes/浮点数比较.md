# C++ 浮点数比较

## 核心结论

浮点数**不能**直接用 `==` 比较,因为 IEEE 754 表示精度有限,任何经过运算的浮点数都可能携带表示误差或累积误差。正确做法是引入 epsilon 容忍度,根据数值范围选择绝对误差、相对误差或二者混合的比较策略。

这不是 C++ 的 bug——Java、Python、JavaScript 同样基于 IEEE 754,问题相同。理解根因需要从二进制表示和 IEEE 754 存储格式出发。

## 一、根因:为什么 `0.1` 不等于 `0.1`

### 二进制表示的先天缺陷

`0.1` 转二进制是 `0.00011001100110011001100110011...`(0011 无限循环)。IEEE 754 单精度 `float` 尾数只有 23 位有效数字(约 7 位十进制精度),存储时必须在 23 位处截断——这个截断就是误差的源头。

```cpp
// 反例:为什么 0.1 + 0.2 != 0.3
double a = 0.1;        // 实际存储:0.10000000000000000555...
double b = 0.2;        // 实际存储:0.20000000000000001110...
double c = a + b;      // 实际存储:0.30000000000000004440...
c == 0.3;              // false,真实是 0.30000000000000004
```

任何"两个带误差的数相加"的运算,误差都会累积。`==` 比较的是**精确值**,而浮点数存储的从来都不是精确值。

### IEEE 754 双精度表示

`double` 用 64 位:1 位符号、11 位指数、52 位尾数,约 15-17 位十进制精度。`float` 用 32 位:1 位符号、8 位指数、23 位尾数,约 7 位十进制精度。精度差异决定了 epsilon 的取值范围。

## 二、三种比较策略

### 绝对误差

```cpp
bool equalAbs(double a, double b, double eps = 1e-9) {
    return std::abs(a - b) < eps;
}
```

- **优点**:简单直观,无分支
- **缺点**:eps 是固定阈值。对大数(`1e6` vs `1e6 + 1e-9`)过严,对极小数(`1e-9` vs `2e-9`)过松
- **适用场景**:数值范围已知且固定(像素坐标 [0, 255]、颜色值 [0, 1])

### 相对误差

```cpp
bool equalRel(double a, double b, double eps = 1e-9) {
    return std::abs(a - b) < eps * std::max(std::abs(a), std::abs(b));
}
```

- **优点**:按比例衡量误差,对不同数量级都公平
- **缺点**:当 `a` 和 `b` 都接近 0 时,分母爆炸(0.0001% 误差会被放大成"无穷大误差")
- **适用场景**:科学计算、跨数量级数据(天文、金融)

### 混合方法(推荐)

```cpp
bool almostEqual(double a, double b,
                 double relEps = 1e-9, double absEps = 1e-12) {
    double diff = std::abs(a - b);
    if (diff <= absEps) return true;                        // 接近 0 直接用绝对误差
    return diff <= relEps * std::max(std::abs(a), std::abs(b));
}
```

绝大多数生产代码采用此模式:先用绝对误差快速过滤 0 附近,再用相对误差处理一般情况。

## 三、四个常见陷阱

### 累积误差

```cpp
double sum = 0.0;
for (int i = 0; i < 1000000; i++) sum += 0.1;
// sum 不会是精确的 100000.0,而是 100000.00000001653...
```

每次加 `0.1` 都引入截断误差,误差在循环中累积。金融计算遇到这种情况必须用定点数(如 `int` 存"分"或 `boost::multiprecision::cpp_dec_float`)。

### 吸收现象

```cpp
1e20 + 1.0 == 1e20;  // true!
```

`1.0` 相对于 `1e20` 太小,在 `double` 的 52 位尾数精度下被"吸收"。这是物理模拟的常见 bug 源:大数 + 小数 = 大数,能量不守恒。

### 抵消现象

```cpp
double a = 1e20;
double b = 1e20 + 1.0;   // b == a (吸收)
double diff = b - a;     // diff 可能是 0,也可能是某个不相关的尾数
```

两个相近大数相减,有效数字急剧减少。算法上应避免"大数减大数",改用代数重写(用 `std::hypot`、Welford 算法等)。

### 非结合性

```cpp
double a = 1e20, b = -1e20, c = 1.0;
(a + b) + c;  // 0 + 1.0 = 1.0
a + (b + c);  // 1e20 + (-1e20 + 1.0) = 1e20 + (-1e20) = 0
```

加法不满足结合律。并行计算中,如果求和顺序与串行不一致,结果可能不同。OpenMP/MPI 的 `reduce` 需特别留意。

## 四、什么时候可以直接用 `==`

极少数合法场景:

```cpp
double x = 1.0;          // 字面量赋值,无运算
if (x == 1.0) { ... }    // OK,实际比较的是两个相同的 IEEE 754 比特位

double y = a;            // 同一赋值,无运算
if (x == y) { ... }      // OK

if (x == 0.0) { ... }    // OK,检查零值
```

只要值经过任何运算(加、减、乘、除、`sin`、`exp`...),就**不能**用 `==`。判断 NaN 也是例外场景:

```cpp
double x = 0.0 / 0.0;    // NaN
x == x;                  // false(NaN 的特殊行为)
std::isnan(x);           // true,推荐写法
```

## 五、epsilon 怎么选

没有万能答案,关键看**业务精度需求**和**数据范围**:

| 数据类型 | 绝对误差典型值 | 相对误差典型值 | 备注 |
|---------|--------------|--------------|------|
| `float`  | 1e-5 ~ 1e-6 | 1e-5 ~ 1e-6 | ~7 位有效数字 |
| `double` | 1e-9 ~ 1e-12 | 1e-12 ~ 1e-15 | ~15-16 位有效数字 |
| `long double` | 1e-12 ~ 1e-15 | 1e-15 ~ 1e-18 | 平台相关(80 位或 128 位) |

参考值:`std::numeric_limits<double>::epsilon()` ≈ 2.22e-16(这是"1.0 与下一个可表示 double 的差",是 epsilon 选择的**下限**,实际工程 epsilon 通常比它大 1-3 个数量级)。

## 六、面试高频追问

**Q1: 0.1 + 0.2 不等于 0.3 是 C++ 的 bug 吗?**
不是。这是 IEEE 754 的标准行为,任何用浮点的语言(Java/Python/JS)都有此问题。Python 给出 `0.30000000000000004`,JavaScript 给出 `0.30000000000000004`,原因完全相同。

**Q2: 什么时候用 `std::numeric_limits<T>::epsilon()`?**
作为相对误差的下界参考。实际工程 epsilon 通常取它的 1-1000 倍(取决于可接受的容差),不要直接用 `epsilon()` 作为绝对误差——它只对 1.0 附近的数有意义。

**Q3: 金融/账务系统为什么不用 `double`?**
因为累积误差无法接受。`double` 在加减 1e6 次后误差可能达到 1e-10 量级,对"分"为单位的钱来说就是 0.001 分的偏差。解决方案:用定点数(`int` 存"分")或任意精度库(`boost::multiprecision::cpp_dec_float_50`)。

**Q4: NaN 为什么 `x == x` 是 false?**
IEEE 754 规定 NaN 与任何值(包括自己)的所有比较都返回 false。这是 C++ 标准沿用的设计,目的是让"传染性错误"可被检测到。如果 `0.0/0.0` 返回 `NaN`,后续所有涉及 NaN 的运算都是 NaN,错误会"传染"到最终结果而不是被静默吞掉。
