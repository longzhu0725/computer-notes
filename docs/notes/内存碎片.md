# C++ 内存碎片治理

## 核心结论

避免内存碎片的核心策略是**减少不同生命周期、不同大小内存块的交替申请与释放**。从最朴素到最专业的方案依次递进:**能不分配就不分配 → 连续分配 → 自己管理分配**。

| 碎片类型 | 描述 | 示例 |
|---------|------|------|
| **外部碎片** | 空闲内存总和够用,但分散在不连续小块,无法满足大分配 | 堆上零散 A/B/C 三个对象,中间有空隙,放不下 D |
| **内部碎片** | 分配的块比请求大,多出的部分浪费 | `malloc` 对齐多给字节,定长对象池块大于实际对象 |

## 一、五层防御体系(从轻到重)

### 第一层:不分配

```cpp
// 优先用栈、静态存储、成员变量
void processData() {
    int buf[1024];                    // 栈,无碎片
    static std::vector<int> cache;    // 静态,只分配一次
    // ...
}
```

栈分配速度极快(只是移动栈指针),且完全无碎片。函数返回时自动释放,**不参与堆的碎片竞争**。

### 第二层:用 STL 容器管理连续内存

```cpp
// vector 内部是连续内存,极少外部碎片
std::vector<MyObj> v;
v.reserve(1000);  // 预分配,避免多次扩容
for (int i = 0; i < 1000; ++i) {
    v.emplace_back(args);  // 不会触发重新分配
}
```

**关键技巧**:`reserve()` 预分配容量,避免扩容时拷贝旧元素。`std::string`、`std::array` 同理。

### 第三层:智能指针防泄漏

```cpp
// RAII 自动管理,杜绝"忘记 delete"导致的不可回收碎片
auto p = std::make_unique<MyObj>();
// p 析构时自动释放内存,不留永久垃圾
```

智能指针的贡献是**间接的**——它防止内存泄漏,而泄漏的内存对系统来说就是"永远无法使用的碎片"。

### 第四层:对象池(Object Pool)

```cpp
class NodePool {
    struct Node { int data; Node* next; };
    std::vector<Node> storage_;  // 一次性分配大块
    Node* freeList_ = nullptr;
public:
    Node* acquire() {
        if (!freeList_) {
            storage_.emplace_back();
            return &storage_.back();
        }
        Node* n = freeList_;
        freeList_ = freeList_->next;
        return n;
    }
    void release(Node* n) {
        n->next = freeList_;
        freeList_ = n;
    }
};
```

适用场景:
- 高频申请/释放的**固定大小**小对象(链表节点、网络包、粒子)
- 长生命周期系统(避免运行时 `new/delete` 抖动)
- 实时性要求高的场景(无 `malloc` 不可预测延迟)

**优势**:
- 分配/归还 O(1)(只是链表头插/头删)
- 内存连续(整块 `vector` 内部无碎片)
- 无系统调用(`new/delete` 会触发 `brk`/`mmap`)

### 第五层:自定义分配器(Custom Allocator)

```cpp
template<class T>
class PoolAllocator {
    // 内部用对象池分配 T 大小的块
public:
    T* allocate(size_t n) {
        return static_cast<T*>(pool_.allocate(n * sizeof(T)));
    }
    void deallocate(T* p, size_t) {
        pool_.deallocate(p);
    }
    // ...
};

std::vector<int, PoolAllocator<int>> v;  // 用内存池的 vector
```

`std::vector`、`std::list`、`std::map` 全部支持自定义分配器,替换默认的 `new/delete`。这是 STL 与内存池的桥梁。

## 二、`std::vector` vs `std::list` 的碎片差异

| 容器 | 内存布局 | 外部碎片 | 性能特征 |
|------|---------|---------|---------|
| `std::vector` | 连续 | 极少 | 缓存友好,O(1) 随机访问 |
| `std::list` | 离散节点 | **大量** | 每次插入/删除都 `new` 节点 |

```cpp
// 高频插入/删除的 list:每次都分配/释放一个节点
for (int i = 0; i < 1000000; ++i) {
    lst.push_back(i);  // 100 万次 new
}
// 产生 100 万个离散的小节点,堆上碎片严重

// 用 vector 预分配:
v.reserve(1000000);
for (int i = 0; i < 1000000; ++i) {
    v.push_back(i);  // 0 次额外分配(已预分配)
}
```

**经验法则**:
- 默认用 `std::vector`(`push_back` 均摊 O(1),缓存友好)
- 只有需要稳定迭代器(insert/erase 不失效其他迭代器)时才用 `std::list`
- 高频小对象用 `std::deque`(分段连续)或自定义对象池

## 三、对象池的线程安全

### 方案 1:全局 mutex(简单但有竞争)

```cpp
class ThreadSafePool {
    std::mutex m_;
    Node* freeList_ = nullptr;
public:
    Node* acquire() {
        std::lock_guard<std::mutex> lock(m_);
        return popFreeList();
    }
    void release(Node* n) {
        std::lock_guard<std::mutex> lock(m_);
        pushFreeList(n);
    }
};
```

简单但所有线程争用同一把锁,高并发时性能差。

### 方案 2:线程本地存储(TLS)+ 全局池兜底(高性能)

```cpp
thread_local Node* tlsFreeList_ = nullptr;
thread_local std::vector<Node> tlsStorage_;

Node* acquire() {
    if (tlsFreeList_) {
        Node* n = tlsFreeList_;
        tlsFreeList_ = tlsFreeList_->next;
        return n;  // 0 锁,极快
    }
    // 本地池耗尽,从全局池批量获取
    return acquireFromGlobal();  // 偶尔加锁
}
```

每个线程有自己的空闲列表,99% 的操作无锁。本地池耗尽时再访问全局池(批量获取,减少锁次数)。

## 四、自定义分配器的实战价值

### 性能对比(典型场景)

| 分配方式 | 100 万次 `new/delete` 耗时 | 碎片程度 |
|---------|--------------------------|---------|
| 默认 `new/delete` | 200-500 ms | 严重 |
| `std::vector` 预分配 | 5-10 ms | 几乎无 |
| 内存池 | 1-5 ms | 无 |

差距可达 **100 倍**——这是高频交易、游戏服务器等性能敏感场景必须优化的点。

### 库支持

- `boost::pool`:通用对象池
- `jemalloc` / `tcmalloc`:替代 `glibc malloc`,减少多线程碎片
- `std::pmr::polymorphic_allocator`(C++17):标准多态分配器接口

## 五、面试高频追问

**Q1: 外部碎片和内部碎片举例?**
- **外部**:堆上 A/B/C 三个对象,中间有空闲,但放不下 D(因为 D 需要 5 个连续块,而空闲分散)
- **内部**:`malloc` 内存对齐多给 8 字节,或定长对象池的块大小是 64 字节但你申请的对象只有 48 字节,剩下 16 字节浪费

**Q2: `std::list` 和 `std::vector` 在碎片上的区别?**
`vector` 单块连续,无外部碎片,扩容时可能产生暂时碎片。`list` 每个节点独立分配,频繁插入/删除产生**大量**外部碎片。高频小对象场景应避免 `list`,改用 `vector` 预分配或对象池。

**Q3: 智能指针怎么帮助减少碎片?**
主要贡献是**防泄漏**。泄漏的内存对系统就是"永远无法使用的碎片",智能指针通过 RAII 确保释放,间接对抗泄漏引起的碎片。对碎片本身的减少作用是间接的(它不直接做内存池),但很重要。

**Q4: 线程安全的对象池怎么设计?**
- 简单:mutex 保护 freeList
- 高性能:线程本地存储(TLS),每个线程自己的空闲列表,无锁
- 本地池耗尽时再去访问全局加锁池(批量获取减少锁次数)

**Q5: 实际项目里怎么选择?**
四步走:
1. **默认用 STL 容器 + `reserve()`** —— 解决 80% 场景
2. **智能指针** —— 解决泄漏引起的碎片
3. **对象池** —— 解决高频固定大小小对象(网络包、链表节点)
4. **自定义分配器 + jemalloc** —— 解决极端性能场景
