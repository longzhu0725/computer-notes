# 封装

封装（Encapsulation）是 [C++ 学习路线(2026)](/notes/c++-学习路线2026.html) 之一，指把数据和对数据的操作打包在同一个 [封装](/notes/封装.html) 中，并通过访问控制符约束外部对内部状态的直接修改。其核心思想是**隐藏实现细节、暴露可控接口**，从而让类的内部演进不破坏外部调用方。

## 定义与核心思想

封装由两个机制组成：

- **数据与操作的聚合**：把相关字段和方法放进同一个 [封装](/notes/封装.html)，调用方无需关心数据存放在哪里、以何种结构组织。
- **访问控制**：通过 `public` / `private` / `protected` 三个访问级别，决定哪些成员对外部可见、哪些仅限类内使用、哪些可被派生类访问。

封装的价值不在于"语法上能不能用"，而在于"能否在不破坏调用方的前提下修改实现"。一个没有任何 `private` 成员的类在语法上合法，但等同于结构体，并不构成真正的封装。

## 关键原理

### 访问控制的三级语义

| 修饰符 | 类内 | 派生类 | 外部 | 典型用途 |
|---|---|---|---|---|
| `public` | ✅ | ✅ | ✅ | 对外接口 |
| `protected` | ✅ | ✅ | ❌ | 给派生类使用的扩展点 |
| `private` | ✅ | ❌ | ❌ | 内部状态、辅助函数 |

`struct` 与 `class` 的唯一语法差异是默认访问级别：`struct` 默认 `public`，`class` 默认 `private`。实际项目里，惯例是用 `struct` 表示纯数据集合，用 `class` 表示具有不变式与接口的对象。

### 封装与 [虚函数 vs 纯虚函数](/notes/虚函数-vs-纯虚函数.html) 的区别

两者经常被混用，但层次不同：

- **抽象**回答"对外暴露什么"——设计层面的取舍
- **封装**回答"如何把不暴露的东西藏起来"——实现层面的机制

一个类可以封装得很好但抽象糟糕（暴露了一堆不该暴露的 `public` 方法），也可以抽象清晰但封装脆弱（成员全 `public`，调用方直接修改内部状态）。

### 封装不等于"全部 private"

过度封装的反模式是"getter/setter 一把梭"：

```cpp
class Point {
private:
  double x_, y_;
public:
  double getX() const { return x_; }
  void setX(double x) { x_ = x; }
  // ...
};
```

这种写法在语法上封装了字段，**但没有任何不变式保护**——调用方依然能通过 setter 把 x 设为非法值（如 NaN）。真正有意义的封装应当是暴露**有意义的行为**（如 `translate(dx, dy)`），而不是简单的字段读写。

## 应用与实例

### 最小可运行示例

```cpp
class Thermostat {
public:
  void setTargetCelsius(double c) {
    if (c < 5.0 || c > 40.0) {
      throw std::out_of_range("target out of safe range");
    }
    target_ = c;
  }
  double getTargetCelsius() const { return target_; }

private:
  double target_{20.0};
};
```

调用方拿到的是 `setTargetCelsius` / `getTargetCelsius` 两个接口，无法绕过范围检查直接修改 `target_`。如果未来 `target_` 改为从配置文件读取，或新增了与 [继承](/notes/继承.html) 配合的虚函数，外部调用代码完全不需要修改。

### 典型使用场景

- **领域模型**：业务实体的字段、状态转移、合法性校验
- **资源管理**：RAII 句柄、智能指针、文件句柄
- **库边界**：把内部数据结构隐藏在稳定 API 之后，让库的内部重构不影响下游

## 与其他概念的关系

- [继承](/notes/继承.html) 在派生类中会改变 `protected` / `private` 成员的可见性，但封装层级本身是按类定义的。
- [多态](/notes/多态.html) 要求基类暴露稳定的虚函数接口——这本身就是封装在继承体系中的延伸。
- [C++ 学习路线(2026)](/notes/c++-学习路线2026.html) 三个概念彼此独立，但在设计良好的类层级中通常**同时**存在。
- [虚函数 vs 纯虚函数](/notes/虚函数-vs-纯虚函数.html) 是封装的更高层目标；[虚函数 vs 纯虚函数](/notes/虚函数-vs-纯虚函数.html) 是封装在多态语境下的极端形式（只暴露虚函数，不暴露数据）。
