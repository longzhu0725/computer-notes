# C++ 变量作用域

## 核心结论

理解 C++ 变量需要抓住**三个独立维度**:**作用域**(可见范围)、**存储期**(生命周期)、**链接性**(跨翻译单元可见性)。`static` 关键字在不同位置产生不同效果——这是初学者最容易混淆的地方。

| 变量类型 | 作用域 | 存储期 | 存储位置 | 链接性 | 默认初始化 |
|---------|--------|--------|---------|--------|-----------|
| 局部变量 | 所在块 | 块执行期 | 栈 | 无 | 不确定(栈残留) |
| 静态局部变量 | 所在函数 | 整个程序 | 数据段/BSS | 无 | 零初始化(首次进入时) |
| 全局变量 | 整个文件 | 整个程序 | 数据段/BSS | 外部 | 零初始化 |
| 静态全局变量 | 所在文件 | 整个程序 | 数据段/BSS | 内部 | 零初始化 |

## 一、三个维度的关系

### 作用域(scope):代码中可见的范围

```cpp
int g = 10;             // 文件作用域:整个 .cpp 文件可见
void f() {
    int x = 1;          // 块作用域:只在 f() 内可见
    { int y = 2; }      // y 在此块外不可见
}
```

### 存储期(storage duration):对象存在多久

- **自动存储期**:块执行时创建,退出块时销毁(局部变量、函数参数)
- **静态存储期**:程序启动时创建,程序结束时销毁(全局变量、`static` 变量)
- **动态存储期**:`new` 时创建,`delete` 时销毁(堆对象)
- **线程存储期**:`thread_local` 修饰,线程结束时销毁(C++11)

### 链接性(linkage):跨翻译单元是否可见

- **外部链接**:可在其他 `.cpp` 文件中通过 `extern` 引用
- **内部链接**:仅在当前 `.cpp` 文件可见
- **无链接**:仅在当前作用域可见

## 二、四种变量的内存布局

```
高地址
+------------------+
|      栈(stack)    | ← 局部变量,函数参数,返回地址
|        ↓          |
|                  |
|                  |
|        ↑          |
|     堆(heap)     | ← new/malloc 分配
+------------------+
|     BSS 段        | ← 未初始化/零初始化的全局/静态变量
+------------------+
|   数据段(data)    | ← 已初始化的全局/静态变量
+------------------+
|   代码段(text)    | ← 编译后的机器指令
低地址
```

**关键点**:静态局部变量**不**在栈上,而在数据段/BSS 段。栈帧销毁时,栈上的局部变量消失,但静态局部变量依然存在(下次调用函数时保留上次的值)。

## 三、`static` 的四种用法

C++ 中 `static` 有四种语义,取决于修饰位置:

| 修饰位置 | 效果 | 示例 |
|---------|------|------|
| 函数内局部变量 | 改变存储期(自动 → 静态) | `void f() { static int x; }` |
| 全局变量 | 改变链接性(外部 → 内部) | `static int g;` |
| 类的成员变量 | 所有对象共享一份 | `class A { static int n; };` |
| 类的成员函数 | 无 `this` 指针,只能访问静态成员 | `class A { static void f(); };` |

**最易混淆的一对**:

```cpp
// 用法 1:函数内的 static → 改变存储期
void counter() {
    static int n = 0;  // 整个程序只初始化一次
    n++;
    std::cout << n;
}
counter();  // 输出 1
counter();  // 输出 2(保留上次的值)

// 用法 2:全局变量的 static → 改变链接性
// file1.cpp
static int hidden = 42;     // 内部链接,其他 .cpp 不可见
// file2.cpp
extern int hidden;          // 链接错误:找不到 hidden
```

## 四、初始化规则的差异

### 局部变量(陷阱所在)

```cpp
void f() {
    int x;        // 警告:未初始化,x 是栈上的随机值
    int y = x;    // 严重 bug:y 也是随机值,且 UB
}
```

栈内存只是函数调用时的"借住地",之前调用留下的残留数据。**永远显式初始化局部变量**。

### 全局/静态变量(安全默认值)

```cpp
int g;                // 0
static int s;         // 0
int* p;               // nullptr
double d;              // 0.0
char c;                // '\0'
bool b;                // false
```

C++ 标准保证零初始化——这是编译器和链接器在 `_BSS` 段中预置的(段被 OS 加载时清零)。

### 静态局部变量的"懒初始化"

```cpp
void f() {
    static int n = compute();  // 只在第一次进入 f() 时执行 compute()
    // 后续调用 f() 不会重新初始化
}
```

编译器实际生成的代码等价于:

```cpp
void f() {
    static int n;             // 零初始化
    static bool inited = false;  // 编译器内部标志
    if (!inited) {
        n = compute();
        inited = true;
    }
    // 使用 n
}
```

**C++11 起,这个初始化是线程安全的**——多个线程同时首次进入 `f()`,只有一个线程执行 `compute()`,其他线程阻塞等待。这是 Meyers 单例模式的理论基石。

## 五、单例模式的两种写法对比

### Meyers 单例(函数内静态变量,推荐)

```cpp
class Singleton {
public:
    static Singleton& get() {
        static Singleton instance;  // 首次调用时构造,线程安全
        return instance;
    }
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;
private:
    Singleton() = default;
};
```

### 全局对象(不推荐)

```cpp
Singleton& get() {
    static Singleton& instance = *global_ptr;  // 依赖全局变量
    return instance;
}
Singleton* global_ptr = nullptr;  // 必须在 main 之前初始化
```

全局对象有**静态初始化顺序问题**——不同翻译单元的全局变量初始化顺序是未定义的。

## 六、静态初始化顺序问题(Static Initialization Order Fiasco)

```cpp
// file1.cpp
Config& getConfig() {
    static Config instance;  // 首次调用时构造
    return instance;
}

// file2.cpp
Logger& getLogger() {
    static Logger instance;  // 首次调用时构造
    return instance;
}
```

`getConfig()` 和 `getLogger()` 谁先被调用?——**未定义**。如果 `Logger` 构造时需要 `Config`,而 `Config` 还未构造,程序崩溃。

**解决方案**:**Construct On First Use** 惯用法(用函数内静态变量替代全局变量)。上面的 Meyers 单例就是例子——静态局部变量在第一次被访问时才初始化,且只初始化一次。

## 七、同名遮蔽与 `::` 解析

```cpp
int value = 100;  // 全局变量

void f() {
    int value = 50;   // 局部变量,遮蔽全局变量
    std::cout << value;     // 50(局部)
    std::cout << ::value;   // 100(显式访问全局)
}
```

**最佳实践**:**避免同名**。遮蔽是 bug 温床,`::` 解析是"补救措施"而不是"推荐做法"。

## 八、面试高频追问

**Q1: 静态局部变量的初始化是线程安全的吗?**
C++11 标准明确保证是。如果多个线程同时首次进入包含 `static` 局部变量的函数,只有一个线程执行初始化,其他线程阻塞等待(由编译器在底层用 `std::mutex` 或原子操作实现)。这让 Meyers 单例在 C++11 后成为线程安全的最简实现。

**Q2: 全局变量初始化顺序问题如何解决?**
"Static Initialization Order Fiasco"指不同翻译单元的全局变量初始化顺序未定义。解法:**Construct On First Use**——把全局变量替换为函数内静态局部变量,把"何时初始化"的控制权交给第一次调用者,而不是链接器。

**Q3: 为什么推荐"能用局部变量就不要用全局变量"?**
四个原因:1) 任何地方都能修改,状态变化难以追踪;2) 跨文件依赖导致强耦合;3) 多线程下需要额外同步;4) 初始化顺序不确定。局部变量作用域最小、生命周期最短,最易推理。

**Q4: 静态局部变量和全局变量都存数据段,有什么区别?**
**存储位置**相同(都在数据段/BSS 段),但**作用域**不同。静态局部变量只能在定义它的函数内访问,编译器会阻止外部访问。全局变量(无 `static`)可被其他文件 `extern` 引用。从封装角度,静态局部变量更安全——它把"持久状态"限制在函数内部。
