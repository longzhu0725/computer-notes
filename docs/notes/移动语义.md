# C++ 移动语义

## 核心结论

移动语义让对象**搬家**而不是**克隆**。通过右值引用 `T&&` 识别"对象马上要死了",然后移动构造函数把源对象的内部指针**接管**过来(置空源对象),**O(1)** 完成。原本 `vector<string>` 拷贝要 O(n) 深拷贝,移动后只搬指针。

| 操作 | 拷贝 | 移动 |
|------|------|------|
| `vector<string>` 1000 元素 | O(n):分配 + 1000 次 string 拷贝 | O(1):交换 3 个指针 |
| `string` 10MB 字符 | O(n):分配 + memcpy | O(1):交换 1 个指针 |
| `unique_ptr<T>` | 禁止(语义错误) | O(1):交换 1 个指针 |

## 一、解决了什么问题

### C++11 之前的浪费

```cpp
std::vector<std::string> v;
v.push_back(getString());
// getString() 返回临时 string(右值)
// push_back 拷贝构造新元素:分配新内存 + memcpy(字符)
// 临时 string 析构:释放原内存
// 浪费:临时对象本来马上要死,内存可以直接接管
```

### 移动语义的解法

```cpp
// 移动构造(典型实现)
class string {
    char* data_;
    size_t size_;
public:
    string(string&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;   // 关键:把源对象置空
        other.size_ = 0;
    }
};

// push_back 看到 string&& 重载 → 调移动构造
// 总成本:交换 1 个指针,O(1)
```

## 二、实现原理

### 移动构造的标准写法

```cpp
class Buffer {
    char* data_;
    size_t size_;
public:
    // 移动构造:接管 + 置空
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    // 移动赋值:先释放自己,再接管
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {       // 自赋值检查
            delete[] data_;          // 释放自己旧资源
            data_ = other.data_;
            size_ = other.size_;
            other.data_ = nullptr;   // 源对象置空
            other.size_ = 0;
        }
        return *this;
    }
};
```

**核心三步**:
1. 接管源对象的指针
2. 源对象的指针置空(`nullptr`)
3. 标记 `noexcept`(让 `vector` 等容器放心使用)

### 移动后源对象状态

```cpp
std::string s = "hello";
std::string s2 = std::move(s);
// s 进入"有效但未指定"状态(valid but unspecified)
//
// 可以做:
//   s = "new value";     // 赋新值
//   s.clear();           // 显式清空
//   析构 s;              // 析构(nullptr 安全)
//
// 不能做:
//   std::cout << s;      // 读内容(值未指定)
```

## 三、`std::move` 的本质

### 不移动任何东西

```cpp
template<class T>
constexpr std::remove_reference_t<T>&& move(T&& t) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(t);
}

// std::move(x) 等价于 static_cast<T&&>(x)
// 只是把 x 的类型从左值改成右值引用
// 实际移动发生在 移动构造函数 里
```

### 触发移动的三种方式

```cpp
// 1. 临时对象(自动是右值)
std::vector<std::string> v;
v.push_back(std::string("temp"));  // 触发移动
v.push_back("literal");            // 触发移动(const char* → string&&)

// 2. 显式 std::move
std::string s = "hello";
v.push_back(std::move(s));         // 触发移动(s 变空)

// 3. 函数返回局部对象
std::string make() {
    std::string local = "...";
    return local;  // 触发移动(或 RVO 直接优化掉)
}
```

## 四、编译器自动触发移动的场景

| 场景 | 触发条件 | 触发结果 |
|------|---------|---------|
| 函数返回局部对象 | RVO/NRVO 失败 | 移动构造 |
| `vector` 扩容搬元素 | 移动构造标 `noexcept` | 移动构造 |
| 用临时对象初始化新对象 | 右值 | 移动构造 |
| `push_back(临时对象)` | 实参是右值 | 移动构造 |

**`push_back` 重载解析**:

```cpp
class vector {
    void push_back(const T& v);   // 拷贝:接左值
    void push_back(T&& v);        // 移动:接右值
};

std::string s = "...";
v.push_back(s);                   // s 是左值 → 拷贝
v.push_back(std::move(s));        // 强转右值 → 移动
v.push_back("temp");              // 临时对象 → 移动
v.push_back(makeString());        // 函数返回右值 → 移动
```

## 五、什么时候该实现移动语义

### 管理的资源:**拷贝代价高**时

| 类 | 是否需要移动 | 原因 |
|----|------------|------|
| `int`、`double` 等值类型 | ✗ | 移动和拷贝等价 |
| `std::array<int, N>` | ✗ | 数组元素是平凡类型 |
| `std::vector<T>` | ✓ | 内部有堆指针 |
| `std::string` | ✓ | 内部有堆指针 |
| `std::unique_ptr<T>` | ✓ (只能移动) | 独占所有权,禁止拷贝 |
| 自定义资源类(socket/锁) | ✓ | 关闭资源代价高 |

**判断公式**:`sizeof(T)` 大,或管理堆内存/系统资源 → 必须实现移动;否则交给编译器自动生成(对简单类,移动 = 浅拷贝,可能不够好,需 `= default`)。

### Rule of Zero vs Rule of Five

**Rule of Zero**(推荐):成员都是 RAII 类型(`std::string`、`std::vector`、`std::unique_ptr`)→ 不需要自定义任何特殊成员函数,编译器自动生成的移动/拷贝都对。

**Rule of Five**(需要时):如果定义了**任一**析构、拷贝构造、拷贝赋值,通常需要定义全部 5 个:

```cpp
class Buffer {
    char* data_;
public:
    ~Buffer();                                // 自定义析构
    Buffer(const Buffer&);                    // 拷贝构造
    Buffer& operator=(const Buffer&);         // 拷贝赋值
    Buffer(Buffer&&) noexcept;                // 移动构造
    Buffer& operator=(Buffer&&) noexcept;     // 移动赋值
};
```

## 六、移动语义和 RVO

### RVO:编译器优化,比移动还快

```cpp
std::vector<std::string> make() {
    std::vector<std::string> local;
    local.push_back("...");
    return local;  // RVO:直接在调用方栈帧构造
}

// 调用方
auto v = make();
// 没有"在 make 内部构造 local → 移动到 v"的步骤
// 直接在 v 的位置构造,零开销
```

### 移动是 RVO 失败的兜底

```cpp
std::vector<std::string> make(bool cond) {
    std::vector<std::string> a;
    std::vector<std::string> b;
    if (cond) return a;  // 多个 return 路径,RVO 失败
    else return b;       // → 触发移动
}
```

C++17 强制某些场景必须 RVO(消除拷贝/移动),但多 return 路径、命名局部对象、try-catch 等场景 RVO 仍可能失败——这时移动语义兜底。

## 七、面试高频追问

**Q1: 为什么移动构造要 `noexcept`?**
`vector` 扩容时,如果移动构造可能抛异常,搬一半失败会导致数据损坏(旧元素已搬走,新元素部分构造,无法回滚)。标记 `noexcept` 后,vector 才有强异常安全保证。**没标记 → vector 退化为拷贝构造**(性能损失数倍)。**所有移动操作都加 `noexcept`**,除非能证明确实可能抛异常。

**Q2: 什么时候该给类实现移动语义?**
类管理了**外部资源**(堆内存、文件句柄、socket、锁)且**拷贝代价高**时。如果只有几个 `int` 成员,移动和拷贝没区别,不需要专门写。`unique_ptr` 是典型——禁止拷贝,只允许移动,因为资源所有权必须唯一。

**Q3: `std::move` 之后还能用源对象吗?**
**能用,但不能假设它的值**。标准保证"有效但未指定"状态:可以析构、可以赋新值,不能读内容当有意义数据。实际工程中,move 后**最好不要再用源对象**,除非是给它赋新值。读 s 的内容是 UB 或未定义值。

**Q4: 移动语义和拷贝省略(RVO)什么关系?**
RVO 是编译器优化,直接在调用方栈帧构造返回值对象,**连移动都省了**(零开销)。移动语义是 **RVO 失败时的兜底方案**。C++17 起某些场景 RVO 强制要求,但多 return 路径、命名局部对象、try-catch 等场景 RVO 可能失败,这时移动语义起作用。

**Q5: 移动和拷贝哪个快?**
取决于资源类型:
- 简单值类型(`int`、`double`):一样快(都是寄存器间复制)
- 容器/字符串/智能指针:**移动快得多**(只搬指针 vs 深拷贝)
- 平凡类(只有 int 成员):一样快
- 资源管理类(堆内存/句柄):**移动快 100-1000x**
