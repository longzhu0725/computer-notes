# 智能指针的线程安全

## 核心结论

智能指针的线程安全要分**三个层面**来看:**控制块线程安全、对象访问不安全、指针实例本身的并发写也不安全**。shared_ptr 的引用计数加减是原子操作,但它管理的对象和 shared_ptr 变量本身都需要你自己加锁。

| 层面 | 线程安全 | 备注 |
|------|---------|------|
| 控制块(引用计数) | ✅ | `std::atomic` 实现,标准保证 |
| shared_ptr 实例本身(同一变量) | ❌ | 多线程写需加锁 |
| 管理的对象(成员访问) | ❌ | 智能指针只管生命周期 |
| unique_ptr 实例 | ❌ | 独占语义,move 不是线程安全 |

## 第一层:控制块(引用计数)— 线程安全

shared_ptr 内部的控制块用 `std::atomic` 实现引用计数的加减。多个线程同时拷贝、析构各自持有的 shared_ptr **副本**,引用计数不会出错。这是**标准保证**的。

```cpp
// 安全:每个线程有独立副本
void worker(std::shared_ptr<Data> sp) {  // 按值传递,引用计数 +1
    // 自己的副本,和其他线程无关
}  // sp 析构,引用计数 -1
```

## 第二层:shared_ptr 实例本身 — 不安全

**同一个 shared_ptr 变量**,多个线程同时读没问题,但如果有线程在写(`reset`、赋值、移动),就必须加锁。

因为写操作要同时修改内部指针和调整引用计数,**不是一个原子步骤**:

```cpp
// 不安全:多线程写同一个 sp
std::shared_ptr<Config> sp = std::make_shared<Config>();

void writer() {
    sp = std::make_shared<Config>(load());  // 写操作:删旧 + 构造新
}
void reader() {
    auto& c = *sp;  // 读操作
}
```

**为什么不是原子的**?写过程:

1. 构造新的 `Config` 对象
2. 在新控制块上 `shared_count = 1`
3. 把 `sp` 内部的指针指向新控制块
4. 旧控制块 `shared_count -1`,可能归零触发释放

步骤 3-4 之间,reader 可能看到 `sp` 指向新对象但旧对象还没释放完成——**部分可见**。

## 第三层:管理的对象 — 不安全

智能指针只管**生命周期**,不管对象**内部的并发访问**。多个线程通过各自的 shared_ptr 副本访问同一个对象,读写对象的成员仍然需要 mutex 保护。

```cpp
auto sp = std::make_shared<Counter>();
// 多个线程:
for (int i = 0; i < 1000; ++i) {
    sp->increment();  // 不安全!++ 不是原子的
}
```

**智能指针完全不知道 `Counter::increment` 是什么语义**,它只负责 `Counter` 对象在所有引用消失时被正确释放。

## unique_ptr 的线程安全

unique_ptr 是**独占语义**,不存在共享的场景:

- **读取**:多线程读同一个 unique_ptr 安全(只是值传递)
- **移动**:move 不是线程安全,必须确保同一时刻只有一个线程操作它

```cpp
std::unique_ptr<Resource> g_up;
void init() {
    g_up = std::make_unique<Resource>();  // 写,需加锁
}
Resource* use() {
    return g_up.get();  // 读,安全(单线程读)
}
```

## 引用计数的内存序

| 操作 | 内存序 | 理由 |
|------|--------|------|
| 增加计数 | `memory_order_relaxed` | 不需要同步其他数据,只是计数 |
| 减少计数 | `memory_order_acq_rel` | 减到零时要释放对象,必须保证之前所有线程对对象的写入都对当前线程可见 |
| 弱引用 lock() | `memory_order_acquire` | 必须看到对象构造完成 |

**为什么 acq_rel?** 当 `shared_count` 减到 0 时,当前线程要释放对象。如果不 acquire,可能看到 `shared_count == 0` 但对象的字段还"未被其他线程写完",导致 use-after-free。

## 怎么安全地在线程间传递 shared_ptr

### 方案 1:值传递(推荐)

```cpp
void worker(std::shared_ptr<Data> sp) {  // 按值接收,各线程独立副本
    sp->process();
}
std::thread t1(worker, std::make_shared<Data>(1));
std::thread t2(worker, std::make_shared<Data>(2));
```

每个线程有自己的副本,引用计数原子 +1,互不干扰。

### 方案 2:C++20 atomic&lt;shared_ptr&gt;

```cpp
std::atomic<std::shared_ptr<Config>> g_cfg;

void reader() {
    auto sp = g_cfg.load();  // 原子读,获得一个 shared_ptr 副本
    use(*sp);
}
void writer() {
    g_cfg.store(std::make_shared<Config>(load()));  // 原子写
}
```

C++20 之前只能用 mutex。

### 方案 3:mutex 保护

```cpp
std::shared_ptr<Config> g_cfg;
std::mutex mtx;

void reader() {
    std::shared_ptr<Config> sp;
    {
        std::lock_guard<std::mutex> lock(mtx);
        sp = g_cfg;  // 加锁读
    }
    use(*sp);
}
```

## 多线程同时读同一个 shared_ptr 变量安全吗?

**安全**。标准规定对同一个 shared_ptr 实例的并发**只读**访问不构成数据竞争。但只要有一个线程在写,**所有访问都必须同步**。

```cpp
const auto& sp = g_cfg;  // 安全:只读引用
```

## 为什么不把 shared_ptr 整体做成原子的?

**性能代价太大**。绝大多数场景下 shared_ptr 是**线程局部**使用的,每次操作都走原子指令会白白浪费性能。

C++20 的 `atomic<shared_ptr>` 是给真正需要**跨线程共享同一个指针变量**的场景准备的,按需使用。

## shared_ptr 的 make_shared 与线程安全

`make_shared<T>(args...)` 在多线程下是**原子**的:整个对象的构造+控制块的创建是原子的(在 `make_shared` 内部完成),不会出现"对象构造到一半被另一个线程看到"。

但**不**意味着返回的 shared_ptr 实例是线程安全的——只是**构造**这一步安全,后续读写仍需同步。

## 实际工程建议

| 场景 | 推荐做法 |
|------|---------|
| 跨线程共享同一个 shared_ptr 变量 | mutex 或 `atomic<shared_ptr>` |
| 跨线程传递 shared_ptr | 值传递 |
| 跨线程访问对象成员 | mutex 保护对象 |
| unique_ptr 跨线程 | 几乎不用,改用 shared_ptr |
| 全局 unique_ptr | 加 mutex 保护,或改成 shared_ptr |

## 面试高频追问

**Q1: 智能指针的引用计数为什么要原子?**
因为多个线程可能同时增减引用计数(如 `std::thread` 持有 shared_ptr 然后退出)。非原子操作会导致计数错乱(丢失更新、读到中间值),引发双重释放或内存泄漏。

**Q2: 智能指针析构是线程安全的吗?**
**析构操作本身**原子(引用计数 -1 是 atomic),但**析构对象的过程**不是线程安全的——如果多个 shared_ptr 共享同一个对象,析构只发生在最后一个引用消失时,由单个线程执行,无并发问题。

**Q3: 怎么安全地从父线程给子线程传 shared_ptr?**
**值传递**(推荐)或 `std::move` 给子线程的入口函数:

```cpp
auto sp = std::make_shared<Data>();
std::thread t([sp = std::move(sp)]() {
    sp->process();
});
t.join();
```

**不要**把父线程的 `sp` 裸指针传给子线程——子线程没有引用计数,父线程析构后对象会释放。

**Q4: weak_ptr 的 lock() 是线程安全的吗?**
**安全**。lock() 内部原子地读 `shared_count`,如果 &gt; 0 则增加计数并返回 shared_ptr(原子 acquire 操作)。但**lock() 之后的对象访问仍需用户自己同步**。
