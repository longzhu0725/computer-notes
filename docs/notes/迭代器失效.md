# C++ STL 迭代器失效

## 核心结论

**迭代器失效 = 迭代器指向的内存地址不再有效**(类似悬垂指针)。失效后继续使用是 UB——可能崩溃,也可能读到垃圾数据。**根本原因有三**:内存重新分配(`vector` 扩容)、元素位置移动(中间删除导致后续元素前移)、节点被销毁(`list`/`map` 删除节点)。**核心规律**:**连续内存容器(vector)失效最严重,链表/树结构容器失效最小**。

| 容器 | 插入 | 删除 |
|------|------|------|
| `vector` / `string` | 扩容:全部失效;否则插入点之后失效 | 被删点及之后全部失效 |
| `deque` | 首尾:迭代器有效但引用可能失效;中间:全部失效 | 首尾:仅被删;中间:全部失效 |
| `list` / `forward_list` | **永不失效** | 仅被删元素失效 |
| `set` / `map` | **永不失效** | 仅被删元素失效 |
| `unordered_*` | rehash:全部失效;否则不失效 | 仅被删元素失效 |

## 一、根因理解

### 三个失效来源

```cpp
// 1. 内存重新分配(扩容)
std::vector<int> v = {1, 2, 3};
auto it = v.begin();   // it 指向旧内存
v.push_back(4);        // 触发扩容 → 旧内存被释放
*it;                    // UB!it 指向已释放的内存

// 2. 元素位置移动(中间删除)
std::vector<int> v = {1, 2, 3, 4, 5};
auto it = v.begin() + 2;  // 指向 3
v.erase(v.begin());       // 删除 1,后续元素前移
*it;                       // 现在指向 4(地址变了!)

// 3. 节点被销毁
std::list<int> lst = {1, 2, 3};
auto it = lst.begin();   // 指向 1
auto next = lst.erase(it);  // 删除 1 节点
*it;                       // UB!节点被 free
// 但 next 仍指向 2(其他节点地址不变)
```

## 二、逐容器失效规则详解

### `vector` / `string`(连续内存)

```cpp
std::vector<int> v = {1, 2, 3, 4, 5};
auto it1 = v.begin();        // 指向 1
auto it2 = v.begin() + 3;    // 指向 4

// 插入
v.insert(v.begin() + 2, 99);
// 不触发扩容:it1 不变, it2 仍指向 4? 错!
// 实际上:插入点之后的 it2 失效(it2 = 指向 4,但 4 现在在 +3 的位置,仍指向 4 的内容但容器内位置变了)
// 严格:插入点之后所有迭代器失效
// 插入点之前的 it1 仍有效

// 触发扩容(超过 capacity):
v.reserve(5);
v.insert(v.end(), 99);  // 容量 5,大小 5,触发扩容
// 所有迭代器全部失效

// 删除
v.erase(v.begin() + 2);  // 删除 99(原 99 在 +2)
// 删除点之后所有迭代器失效(元素前移)
// 删除点之前的迭代器仍有效
```

**最严格规则**:
- 插入 → 插入点之后全部失效
- 触发扩容 → **全部**失效
- 删除 → 删除点之后全部失效

### `deque`(分段连续)

```cpp
std::deque<int> dq = {1, 2, 3, 4, 5};

// 头尾插入
dq.push_front(0);   // 迭代器仍有效(分段的边界可能重排)
//                  // 但元素引用可能失效

// 中间插入
dq.insert(dq.begin() + 2, 99);  // 全部迭代器失效(可能触发段重排)

// 头尾删除
dq.pop_front();     // 仅被删元素迭代器失效

// 中间删除
dq.erase(dq.begin() + 2);  // 全部迭代器失效
```

**规则总结**:
- 头尾插入/删除:迭代器有效,引用可能失效
- 中间插入/删除:全部失效

### `list` / `forward_list`(链表)

```cpp
std::list<int> lst = {1, 2, 3, 4, 5};
auto it1 = lst.begin();
auto it2 = std::next(lst.begin());

// 插入:永不失效
lst.insert(it1, 0);  // it1, it2 仍有效!

// 删除:仅被删的失效
auto next = lst.erase(it1);  // it1 失效, it2 仍有效
```

**链表优势**:插入/删除只影响被操作的节点,其他节点地址完全不动。

### `set` / `map`(红黑树)

```cpp
std::map<int, int> m = {{1, 1}, {2, 2}, {3, 3}};
auto it = m.find(1);

// 插入:永不失效
m.insert({4, 4});  // it 仍有效

// 删除:仅被删的失效
m.erase(it);  // it 失效,其他迭代器仍有效
```

红黑树的平衡操作不改变已有节点地址——`std::map` / `std::set` 是迭代器最稳定的容器。

### `unordered_*`(哈希表)

```cpp
std::unordered_map<int, int> um;
um.reserve(100);  // 预分配避免 rehash

auto it = um.find(1);

// 插入
um.insert({2, 2});  // 没触发 rehash → it 仍有效
// 如果插入后 size > 100,触发 rehash → it 失效

// 删除
um.erase(it);  // 仅被删失效
```

**规则**:触发 rehash 全部失效(罕见,因为 reserve 之后很少触发)。

## 三、循环删除元素的正确写法

### 错误写法

```cpp
std::vector<int> v = {1, 2, 3, 4, 5};
for (auto it = v.begin(); it != v.end(); ++it) {
    if (*it % 2 == 0) {
        v.erase(it);  // it 失效!
    }
    // ++it:UB!(it 已悬空)
}
```

### 修正:用 erase 返回值

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (*it % 2 == 0) {
        it = v.erase(it);  // erase 返回下一个有效迭代器
    } else {
        ++it;
    }
}
```

### 更优:erase-remove 惯用法

```cpp
// 一次遍历 + 一次 erase
v.erase(std::remove_if(v.begin(), v.end(), 
    [](int x) { return x % 2 == 0; }), 
    v.end());
// std::remove_if 把要删除的元素"挪到末尾",erase 统一删除尾部
// 比循环 erase 快 2-3 倍(单次遍历,单次搬移)
```

## 四、`vector` 扩容的失效细节

```cpp
std::vector<int> v = {1, 2, 3, 4, 5};
v.reserve(5);  // 容量 5,大小 5

auto it = v.begin() + 2;  // 指向 3
std::cout << *it;          // 3

v.push_back(6);  // 触发扩容(大小 6 > 容量 5)
std::cout << *it;  // UB!it 指向旧内存(已被 free)
```

**为什么全部失效**?
扩容流程:
1. 分配新内存(原容量 × 2)
2. 移动所有元素到新内存
3. 释放旧内存
4. 更新内部指针

旧内存被 `delete[]`,所有指向它的迭代器瞬间悬垂。

**避免方法**:
```cpp
v.reserve(N);  // 预分配足够容量
v.push_back(...);  // 永不扩容 → 迭代器稳定
```

## 五、多线程下的失效

### 标准库不是线程安全的

```cpp
std::vector<int> v = {1, 2, 3};
// 线程 A 遍历
for (auto it = v.begin(); it != v.end(); ++it) { ... }

// 线程 B 同时插入
v.push_back(4);

// 这是数据竞争 → UB
// 即使不考虑迭代器失效,只读 + 写同时发生也是 UB
```

### 解决方案

```cpp
// 1. 互斥锁
std::mutex m;
{
    std::lock_guard<std::mutex> lock(m);
    // 读 / 写 操作
}

// 2. 读写锁(读多写少)
std::shared_mutex sm;
{
    std::shared_lock lock(sm);  // 多读者
    // 读
}
{
    std::unique_lock lock(sm);  // 写者独占
    // 写
}

// 3. 收集要改的,最后批量改(无锁)
std::vector<int> to_remove;
for (auto it = v.begin(); it != v.end(); ++it) {
    if (need_remove(*it)) to_remove.push_back(*it);
}
for (int x : to_remove) v.erase(...);

// 4. TBB 等并发容器(内部细粒度锁)
tbb::concurrent_unordered_map<int, int> cm;
```

## 六、调试迭代器失效

### 编译期检查

```cpp
// 1. 替换容器为 debug 版本(GCC)
#define _GLIBCXX_DEBUG
std::vector<int> v;
auto it = v.begin();
v.push_back(1);
*it;  // 抛 __gnu_debug::_Safe_iterator

// 2. MSVC Debug 模式默认开启迭代器检查

// 3. AddressSanitizer(检测内存访问错误)
g++ -fsanitize=address
// 失效的迭代器解引用会被捕获为越界访问
```

### 常见 bug 模式

```cpp
// bug 1:range-for 中删除
for (auto& x : v) {
    if (cond(x)) v.erase(&x);  // UB!range-for 内部迭代器失效
}

// bug 2:循环中递增失效迭代器
for (auto it = v.begin(); it != v.end(); ++it) {
    if (cond) v.erase(it);
    // erase 后 it 失效,++it 是 UB
}

// bug 3:多线程同时读写
```

## 七、面试高频追问

**Q1: vector 和 list 失效规则的核心区别?**
- **vector**:插入/删除影响范围广(扩容全部失效,中间操作影响之后)
- **list**:插入永不失效,删除仅影响被删元素
- 根因:vector 是连续内存,牵一发动全身;list 是离散节点,只影响被操作节点

**Q2: 循环中删除 vector 元素的正确写法?**
- 错误:`for(it=v.begin(); ...){ v.erase(it); ++it; }`——erase 后 it 失效
- 正确:`it = v.erase(it)`,不满足条件才 `++it`
- 更优:erase-remove 惯用法,一次遍历搞定

**Q3: vector 扩容为什么导致所有迭代器失效?**
扩容 = 分配新内存 + 移动所有元素 + 释放旧内存。旧内存被 `delete[]`,所有指向它的指针瞬间悬垂。**预防**:`reserve(N)` 预分配,避免扩容。

**Q4: 多线程下迭代器会失效吗?**
不是"失效"问题,是**数据竞争**——同时读写同一容器,即使不考虑迭代器也 UB。**必须加锁保护**。详见 [多线程与锁](./多线程与锁.md)。

**Q5: 如何调试迭代器失效?**
- GCC: `#define _GLIBCXX_DEBUG` 触发运行时检查
- MSVC: Debug 模式默认开启
- AddressSanitizer: `-fsanitize=address` 捕获内存越界
