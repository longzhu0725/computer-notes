# C++11 多线程与锁

## 核心结论

C++11 通过四个头文件提供完整的多线程支持:`<thread>`、`<mutex>`、`<condition_variable>`、`<atomic>`。**优先用 RAII 锁管理器**(`lock_guard` / `unique_lock`),而不是手动 `lock()` / `unlock()`——后者在异常路径下会死锁或泄漏。

| 组件 | 头文件 | 用途 |
|------|--------|------|
| `std::thread` | `<thread>` | 创建和管理线程 |
| `std::mutex` | `<mutex>` | 互斥锁(不可重入) |
| `std::lock_guard` | `<mutex>` | 轻量 RAII 锁(构锁/析解) |
| `std::unique_lock` | `<mutex>` | 灵活 RAII 锁(支持条件变量) |
| `std::condition_variable` | `<condition_variable>` | 等待-通知机制 |
| `std::atomic<T>` | `<atomic>` | 无锁原子操作 |

## 一、线程管理

### 启动线程的四种方式

```cpp
std::thread t1(func);                  // 1. 函数指针
std::thread t2([](int x) { ... }, 42); // 2. Lambda
std::thread t3(&Class::method, obj);   // 3. 成员函数
std::thread t4(std::move(task));       // 4. 移动的 std::function
```

### 必须 join 或 detach

```cpp
void f() {
    std::thread t(someWork);
    // 错误!t 析构时未 join/detach → std::terminate
    if (condition) return;  // 这里 t 析构,程序崩溃
    t.join();
}
```

**正确做法**:RAII 包装,确保任何退出路径都 join:

```cpp
class ScopedThread {
    std::thread t_;
public:
    template<typename... Args>
    ScopedThread(Args&&... args) : t_(std::forward<Args>(args)...) {}
    ~ScopedThread() { if (t_.joinable()) t_.join(); }
    ScopedThread(const ScopedThread&) = delete;
    ScopedThread(ScopedThread&&) = default;
};
```

或者用 C++20 的 `std::jthread`(自动 join)。

### detach 的使用场景

detach 让线程**后台运行**,无法再 join。典型场景:
- fire-and-forget 任务
- 长期运行的后台服务线程

**警告**:detach 的线程访问的对象必须**比线程活得久**,否则悬空引用。

## 二、互斥锁体系

### 三种 mutex 选型

| 类型 | 特性 | 典型场景 |
|------|------|---------|
| `std::mutex` | 不可重入,最快 | 通用互斥 |
| `std::recursive_mutex` | 同一线程可重入 | 递归函数中加锁 |
| `std::timed_mutex` | 支持 `try_lock_for` | 避免永久阻塞 |

**重入风险**:

```cpp
std::recursive_mutex m;

void recursiveCall(int depth) {
    std::lock_guard<std::recursive_mutex> lock(m);  // OK,递归加锁
    if (depth > 0) recursiveCall(depth - 1);
}
// 用 std::mutex 会死锁!
```

**为什么尽量避免重入**:递归锁会让"哪一层持锁"变得不清晰,容易设计出错误的同步逻辑。优先重构代码消除递归持锁需求。

### 手动 lock 的危险性

```cpp
std::mutex m;
m.lock();
// ... 中间逻辑可能抛异常
doSomething();  // 抛异常 → m 永远不解锁 → 死锁
m.unlock();
```

**正确做法**:永远用 RAII 包装。

## 三、RAII 锁管理器

### `lock_guard` vs `unique_lock`

```cpp
// lock_guard:轻量,够用就用它
{
    std::lock_guard<std::mutex> lock(m);
    // 临界区
}  // 析构自动解锁

// unique_lock:灵活,条件变量必用
{
    std::unique_lock<std::mutex> lock(m);
    // 临界区
    lock.unlock();  // 可以中途解锁
    doOtherStuff();
    lock.lock();    // 可以再次加锁
}  // 析构自动解锁
```

| 特性 | `lock_guard` | `unique_lock` |
|------|-------------|--------------|
| 构造即加锁 | ✓ | ✓(也可 `defer_lock`) |
| 析构即解锁 | ✓ | ✓ |
| 中途 `unlock()` | ✗ | ✓ |
| 配合条件变量 | ✗ | ✓ |
| 移动语义 | ✗ | ✓(可转移所有权) |
| 性能开销 | 零(零开销抽象) | 略高(内部有 flag) |
| 内存占用 | 1 个 mutex* | mutex* + bool flag |

**经验法则**:
- 普通临界区 → `lock_guard`
- 条件变量、延迟加锁、转移所有权 → `unique_lock`

## 四、条件变量:生产者-消费者

### 经典模式

```cpp
#include <condition_variable>
#include <mutex>
#include <queue>

std::queue<int> q;
std::mutex m;
std::condition_variable cv;
bool done = false;

// 消费者
void consumer() {
    while (true) {
        std::unique_lock<std::mutex> lock(m);
        cv.wait(lock, []{ return !q.empty() || done; });
        // 醒来后条件已满足(可能虚假唤醒,需 while 循环重检)
        while (!q.empty()) {
            int v = q.front(); q.pop();
            // 处理 v
        }
        if (done && q.empty()) return;
    }
}

// 生产者
void producer() {
    for (int i = 0; i < 100; ++i) {
        {
            std::lock_guard<std::mutex> lock(m);
            q.push(i);
        }
        cv.notify_one();  // 唤醒一个等待者
    }
    {
        std::lock_guard<std::mutex> lock(m);
        done = true;
    }
    cv.notify_all();  // 唤醒所有
}
```

### 为什么必须 `unique_lock`?

`wait()` 内部要做两件事:
1. **释放锁**(让其他线程能修改条件,如生产者放入数据)
2. **挂起线程**(等待 `notify_*`)
3. **被唤醒后重新加锁**(继续操作共享数据)

`lock_guard` 不支持中途释放锁,只有 `unique_lock` 能配合 `wait()`。

### 为什么 `wait` 要带谓词?

```cpp
cv.wait(lock);              // 错误:可能虚假唤醒(spurious wakeup)
cv.wait(lock, []{return cond;}); // 正确:醒来后检查条件
```

POSIX 标准允许条件变量在没收到 `notify` 时也唤醒线程(由于实现细节,如多核同步)。**永远带谓词**重检条件。

## 五、原子操作

### `std::atomic<T>` 适用场景

```cpp
std::atomic<int> counter{0};  // 计数器,无需 mutex

void increment() {
    counter.fetch_add(1, std::memory_order_relaxed);
}

// 多个线程并发 increment,counter 最终值正确
```

### 原子操作的局限

`std::atomic` 只保证**单个变量**的原子性,不保证**多个变量之间的复合操作**的原子性:

```cpp
// 错误:两个 atomic 不构成原子操作
void transfer(Account& from, Account& to, int amount) {
    if (from.balance.load() >= amount) {       // ① 读
        from.balance.fetch_sub(amount);         // ② 改
        to.balance.fetch_add(amount);           // ③ 改
    }
    // ① 和 ② 之间可能被其他线程插入,导致余额检查和扣款不一致
}

// 正确:整个复合操作用 mutex 保护
std::mutex m;
void transfer(Account& from, Account& to, int amount) {
    std::lock_guard<std::mutex> lock(m);
    if (from.balance >= amount) {
        from.balance -= amount;
        to.balance += amount;
    }
}
```

### 内存序(`memory_order`)

```cpp
counter.fetch_add(1, std::memory_order_relaxed);  // 性能最高,不保证跨变量顺序
counter.fetch_add(1, std::memory_order_acq_rel);  // 较强,跨变量顺序可见
counter.store(42, std::memory_order_release);     // 用于同步"发布"给其他线程
```

**经验法则**:
- 计数器、标志位:`memory_order_relaxed`
- 单个变量的复杂操作:`memory_order_acq_rel`
- 多变量同步发布(锁实现、消息队列):`memory_order_release` + `memory_order_acquire`
- 实在搞不清:**用 `seq_cst`(默认),性能问题再优化**

## 六、死锁的四个避免方法

### 1. 统一加锁顺序

```cpp
// 所有线程都先锁 A 再锁 B
void transfer(Account& a, Account& b) {
    if (&a < &b) { lock(a.m, b.m); }
    else { lock(b.m, a.m); }  // 总是先锁地址小的
}
```

### 2. `std::lock()` 一次性锁多个

```cpp
std::mutex m1, m2;
std::lock(m1, m2);  // 原子地锁两个,不会死锁
std::lock_guard<std::mutex> g1(m1, std::adopt_lock);
std::lock_guard<std::mutex> g2(m2, std::adopt_lock);
```

`std::lock` 内部用 `try_lock` + 回溯算法,保证不产生循环等待。

### 3. RAII 保证异常安全

```cpp
void f() {
    std::lock_guard<std::mutex> lock(m);
    riskyOperation();  // 抛异常也安全,lock 析构解锁
}
```

### 4. `timed_mutex` 加超时

```cpp
std::timed_mutex m;
if (m.try_lock_for(std::chrono::milliseconds(100))) {
    // 拿到锁
} else {
    // 超时,放弃或重试
}
```

## 七、面试高频追问

**Q1: `lock_guard` vs `unique_lock` 区别?**
- `lock_guard`:轻量,构造即锁、析构即解,中间不能控制
- `unique_lock`:灵活,支持延迟加锁(`defer_lock`)、手动 lock/unlock、移动所有权
- 日常用 `lock_guard`,条件变量必须用 `unique_lock`

**Q2: 什么是死锁?怎么避免?**
死锁四条件:互斥、持有并等待、不可剥夺、循环等待。避免方法:统一加锁顺序(打破循环等待)、`std::lock` 一次性锁多个、RAII 保证异常安全、`timed_mutex` 加超时兜底。

**Q3: 条件变量为什么必须配 `unique_lock`?**
`wait()` 内部要:释放锁 → 挂起线程 → 被唤醒后重新加锁。`lock_guard` 不支持中途释放锁,只有 `unique_lock` 能配合。

**Q4: `atomic` vs `mutex` 怎么选?**
- 简单单变量(计数器、标志位)用 `atomic`,零锁开销
- 复合操作(转账:同时改两个余额)必须用 `mutex`,因为 `atomic` 只保证单个操作原子性,不保证多操作一致
- 实际工程中,简单场景也能上锁(代码更清晰,性能损失可接受时)

**Q5: `volatile` 和 `atomic` 有什么区别?**
- `volatile`:阻止编译器优化(每次读都从内存读),**不**提供多线程同步。Java 的 `volatile` 有同步语义,C++ 没有
- `std::atomic`:CPU 层面提供原子读写和内存屏障,**正确**的多线程同步手段
- 错误:`volatile int counter;` 多线程并发++,结果不确定。正确:`std::atomic<int> counter;`
